GAP 01 / P0

协作式发现不应被压缩成“收集几个字段”

置信度：高

现状

ddd-modeling/SKILL.md 已要求参与者、命令、结果、不变式和歧义，但没有明确保留领域事件时间线、策略、热点、矛盾语言与 Example Mapping 反例。

证据

Library 的业务描述来自 Big Picture Event Storming，再经 Example Mapping 与 Design Level；Brandolini 强调矛盾意见可能分别在不同上下文成立，未决问题应标记热点。

建议

增加“发现证据包”：参与者、命令、事件、策略、外部系统、时间边界、热点、示例/反例。只有证据足以支撑立即一致性规则时才提出聚合。

落点

ddd-modeling/SKILL.md Workflow；新增 references/discovery-evidence.md，并在输出契约中加入“证据与热点”。

风险

流程过重会让简单定义题也进入工作坊。应保留现有 direct-answer exception，并按问题复杂度裁剪。

GAP 02 / P0

发现上下文之后，还缺少上下文之间的关系

置信度：高

现状

ddd-modeling/references/bounded-contexts.md 聚焦语言冲突与边界候选；当前输出没有要求识别上游/下游、模型传播、团队依赖或集成模式。

证据

DDD Crew 指出 Context Map 同时描述团队关系、服务提供、模型传播和治理，并建议用小地图回答明确问题。IDDD Samples 通过独立上下文与消息/通知适配器展示集成差异。

建议

在上下文候选稳定后增加可选 Context Mapping 阶段：事实权威、上下游、变更影响、Shared Kernel/ACL/Conformist/Published Language/Open Host Service/Separate Ways。

落点

扩展 ddd-modeling/references/bounded-contexts.md 或新建 context-mapping.md；输出一张只回答当前集成问题的小地图。

风险

模式名称容易遮蔽组织现实。必须先描述关系事实，再给模式标签；不要生成“完整企业地图”。

GAP 03 / P1

六模块模板可能被误读为每个上下文的唯一形状

置信度：高

现状

ddd-project-create 生成统一六模块，README 已说明 sample 不是业务模型；但还没有清楚说明：六模块是部署/团队级基线，不要求每个上下文内部复制六层。

证据

Library 在同一模块化单体中让 Lending 使用六边形、Catalogue 使用 CRUD；事件溯源银行让写侧六边形、读侧按功能组织。局部复杂度决定局部架构。

建议

保留六模块依赖方向，但增加“上下文内部策略”：富领域切片、简单 CRUD 切片、独立读模型切片。模板文档解释它们何时出现，不一次生成全部样板。

落点

java-spring-hex-playbook/references/module-strategies.md 与模板 README；暂不增加模块数量。

风险

过早提供多种模板会造成选择疲劳和生成器膨胀。先写决策指南与两个最小示例，经实际项目验证后再自动化。

GAP 04 / P1

CQRS 需要决策梯度，而不只是读写模型定义

置信度：高

现状

当前持久化与查询参考已分离 Aggregate Repository 与 Application Query Port，但对“何时从同一模型升级到独立模型/独立存储”缺少渐进触发条件。

证据

java-cqrs-intro 明确展示五级演进；Library 只为 Patron Profile/Daily Sheet 创建读模型；事件溯源银行仍允许单聚合查询从事件流满足。

建议

增加 CQRS 决策梯度：分方法 → 分接口 → 分模型 → 异步投影 → 分存储 → 事件溯源。每级要求写出查询压力、一致性窗口、恢复方式与新增运维成本。

落点

java-spring-hex-playbook/references/persistence-and-queries.md，并从 transactions-and-events.md 交叉引用。

风险

“演进路线”可能被误读成成熟度阶梯。必须声明大多数系统应停在前两级，后级不是更高级。

GAP 05 / P1

可靠投递之外，还缺事件顺序、幂等和投影恢复

置信度：高

现状

transactions-and-events.md 已区分 Domain Event、Integration Event、Outbox 与失败语义，tech-messaging.md 覆盖重试/顺序/DLQ；但缺少面向事件模型的统一审查清单。

证据

event-sourcing-cqrs-examples 用版本号处理写侧乐观并发与读侧乱序，并把回放逻辑设计为幂等赋值；Library 用同一业务测试证明同步/异步实现可替换。

建议

新增事件正确性六问：事实所有者、分区/顺序键、消费者幂等键、检查点与重试、投影重建、事件 schema 演进。把 Outbox 放入这条链，而不是作为终点。

落点

java-spring-hex-playbook/references/transactions-and-events.md 为语义入口，tech-messaging.md 承载具体实现与验证命令。

风险

事件溯源项目的要求不能全部施加给普通集成事件。清单需按本地领域事件、可靠集成事件、事件溯源三档裁剪。

GAP 06 / P1

生成器验证了文件，却还没有验证业务架构边界

置信度：中

现状

生成器的 Python 契约测试验证模板路径、占位符、MapStruct/MyBatis 选择与 Maven 构建；示例有领域、应用和适配器测试，但没有 ArchUnit 或完整业务场景。

证据

Library 用 ArchUnit 防止 Domain 依赖 Spring/Infrastructure，并保护上下文依赖；Cargo 项目用生命周期场景测试连接多次用例和异常分支。

建议

分两步加入守护：先生成一个无额外依赖的“架构约束说明 + 测试扩展点”；评审 ArchUnit 依赖后，再加入模块/包依赖测试。补一个跨 Application/Domain/Adapter 的 SampleOrder 场景测试。

落点

ddd-project-create/assets/project-template、template-manifest.json 和 Python 契约测试；变更前需更新 template contract。

风险

ArchUnit 规则容易把当前包名冻结成架构。规则应表达依赖方向和框架隔离，不限制未来按业务切片重组。
