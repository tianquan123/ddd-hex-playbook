00.1 / 核心判断

三句话先带走

这份调研最重要的不是新增几个模式名，而是调整技能从“给答案”到“保留证据、延迟复杂度、执行边界”的方式。

01 / Discovery

先发现，再建模

参与者、命令、事件、策略、热点和反例先于聚合。没有业务证据时，目录和数据库表都不能替代模型。

端口所有权、框架隔离、模块依赖与关键场景要能被测试失败，而不只写在 README。

00.3 / 八条结论

跨项目重复出现的信号

每条都能在至少两个案例或一个案例加权威社区资料中找到证据。

上下文边界不等于部署边界

Library 先以包形成模块化单体，IDDD 用独立项目；业务语言和事实权威才是边界起点。

每个上下文可有不同局部架构

同一系统里，复杂 Lending 用六边形，简单 Catalogue 用 CRUD；读侧也可按功能组织。

聚合由即时不变式约束

Cargo 聚合内同步派生 Delivery，HandlingEvent 跨聚合异步影响它；一致性时机说明边界。

端口属于需求消费者

RoutingService、FindAvailableBook、EventStore 分别由真正需要能力的模型或用例拥有。

CQRS 应渐进采用

先分职责，再分模型；只有读写压力、扩展或组织边界确实需要时才分存储。

事件可靠性不止 Outbox

顺序键、幂等、检查点、投影重建和 schema 演进必须成为同一审查链。

测试承载架构意图

业务规格、用例测试、适配器契约、完整场景和 ArchUnit 分别保护不同层面的承诺。

旧项目可保留新思想

IDDD 的 Java 7 不能直接采用，但上下文自治和投影结构仍有价值；技术年代与设计价值要分开标注。

Bottom line

先增强“如何作出决定”，再增强“自动生成什么”

当前项目已经有清楚的依赖不变量与六模块模板。下一步价值最高的改进，是让模型证据、上下文关系、CQRS/事件条件和架构守护更可执行，而不是继续增加技术专题数量。
