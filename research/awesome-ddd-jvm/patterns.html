02Pattern comparison

Cross-case synthesis

跨项目设计模式

将反复出现的做法还原为稳定原则、团队默认、条件化方案与示例。

02.1 / 图例

同一种 DDD，不同的局部架构

比较的目标不是选出统一目录，而是识别业务复杂度如何驱动设计。四种标签用于阻止样例被误读成规则。

稳定原则

跨项目、跨框架仍成立，违反它通常会伤害模型边界。

团队默认值

没有相反证据时采用，但可由项目政策替换。

条件化方案

只有触发条件、代价与失败恢复明确时采用。

示例

用于说明意图，不把框架、目录或库提升为 DDD 要求。

02.2 / Discovery
稳定原则

Event Storming：代码之前先保存发现证据

项目证据

Library 从 Big Picture 到 Example Mapping、Design Level；ddd-practitioners-ref 把角色、命令、事件、风险点、上下文、示例规格连接到测试骨架。

结论

建模输出不能只剩类名。应保留参与者、命令、成功/失败事件、策略、时间边界、热点和反例；聚合只是这些证据之后的候选。

不要泛化：便利贴颜色和工作坊工具不是模型正确性的证据。

02.3 / Contexts
稳定原则

限界上下文：语言边界先于部署边界

项目证据

Library 先以包形成 Catalogue/Lending 模块化单体；IDDD Samples 用独立项目表达上下文；DDD Crew 补充上下游、ACL、Published Language 等关系。

结论

上下文回答“哪个模型在这里有效、谁对事实有权威”，模块或服务只是实现。每个上下文可按复杂度选择 CRUD、富模型或 CQRS。

不要泛化：一个上下文不必等于一个 Maven 模块或微服务。

02.4 / Aggregates
稳定原则

聚合与不变式：由必须立即成立的规则画边界

项目证据

Cargo 将 RouteSpecification、Itinerary 与 Delivery 放在同一聚合以同步派生状态；HandlingEvent 留在另一聚合。Library 用 AvailableBook、BookOnHold 等类型使非法状态更难表达。

结论

先写“在一次业务操作结束前必须成立什么”，再选聚合。跨聚合引用和最终一致性不是追求小对象，而是承认并发与事务边界。

不要泛化：“一个事务一个聚合”是强默认，不是无需业务证据的绝对戒律。

02.5 / Hexagon
团队默认值

端口与适配器：端口属于真正的消费者

项目证据

DDDSample 的 RoutingService 由领域需要、远程求解器实现；Library 的 FindAvailableBook 位于保留用例附近；事件溯源银行将 EventStore 作为写模型出站端口。

结论

根据需求所有者放置接口，而不是把所有端口塞进 domain。转换发生在跨越边界处，Application 负责用例与事务，适配器处理协议和技术失败。

不要泛化：简单 CRUD 上下文无须为目录对称性复制完整六边形。

02.6 / CQRS
条件化方案

CQRS：从方法分离到存储分离逐级升级

项目证据

java-cqrs-intro 展示五级演进；Library 只在 Patron Profile/Daily Sheet 等查询压力处使用读模型；事件溯源银行仍允许单聚合查询走事件流。

结论

先分命令与查询的职责，再看模型是否被查询扭曲；只有性能、扩展、组织自治或存储需求明确时才分数据库。每一级都要写清同步延迟与恢复成本。

不要泛化：DDD 不要求 CQRS，CQRS 也不要求 Event Sourcing。

02.7 / Events
条件化方案

事件与一致性：Outbox 只是开始

项目证据

Library 用同一行为断言对比强/最终一致；事件溯源银行用版本号解决乐观并发与乱序，并让回放只赋值；IDDD README 主动披露双存储的小窗口。

结论

事件设计至少回答：谁拥有事实、顺序按什么键、消费者如何幂等、失败如何重试、投影从何处重建、事件如何演进。Domain Event 与 Integration Event 必须分开。

不要泛化：本地 Spring Event 不自动具备可靠投递语义。

02.8 / Persistence
团队默认值

持久化与查询：写模型负责规则，读模型负责问题

项目证据

DDDSample 用 Repository 重建聚合并测试 JPA 映射；Library 明确选择无 ORM；IDDD 同时展示 Hibernate、LevelDB、JDBC DTO 与 MySQL 投影。

结论

Repository 的单位是聚合而非表；复杂查询可使用独立 Query Port 或读模型，无需让聚合暴露分页 DTO。缓存是适配器策略，不改变模型所有权。

不要泛化：ORM 或 MyBatis 都不是 DDD 的正反面，关键是技术模型是否反向支配领域边界。

02.9 / Guardrails
团队默认值

测试与架构守护：让“我们采用六边形”可以失败

项目证据

Library 用 ArchUnit 阻止模型依赖 Spring/Infrastructure，并以场景测试验证一致性替换；Cargo 项目用完整生命周期测试连接业务语言、应用服务和适配器。

结论

测试组合应包含：纯领域规则、用例编排、适配器契约、关键事务/失败路径、端到端业务场景、架构依赖规则。目录说明无法替代可执行约束。

不要泛化：架构测试只能守住可静态观察的边界，无法证明聚合划分来自真实业务。
