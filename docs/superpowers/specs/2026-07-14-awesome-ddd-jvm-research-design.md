# Awesome DDD JVM 调研报告设计

## 目标

从 `heynickc/awesome-ddd` 的 Sample Projects、Blogs 与 Community Resources 中筛选对 JVM/Java DDD 实践有直接价值的资料，形成一组可离线浏览的静态 HTML。报告既要保留可追溯的原始链接和代码证据，也要把结论映射到当前仓库的三个技能：`ddd-modeling`、`java-spring-hex-playbook` 与 `ddd-project-create`，为后续优化讨论提供依据。

报告不是完整复制 Awesome DDD，也不是按 star 数排序的项目收藏夹。它优先回答三个问题：优秀样例如何从业务知识形成模型；如何在 JVM 工程中保护边界并处理一致性；当前技能已经覆盖什么、还缺什么。

## 调研范围

### 深入分析

选择 6–8 个 JVM 项目作为核心案例，优先考虑以下候选：

- `ddd-by-examples/library`：真实业务需求、Event Storming、模块化单体、按上下文选择局部架构、CQRS、事件一致性与 ArchUnit。
- `citerus/dddsample-core`：经典 Cargo 模型、通用语言、聚合与领域服务的基准实现。
- `eclipse-ee4j/cargotracker`：同一经典领域在 Jakarta EE 中的端到端企业实现。
- `VaughnVernon/IDDD_Samples`：多个限界上下文、上下文集成、CQRS 与事件溯源参考。
- `asc-lab/java-cqrs-intro`：从无 CQRS 到命令查询分离、独立模型、独立存储与事件溯源的渐进路径。
- `andreschaffer/event-sourcing-cqrs-examples`：事件顺序、幂等、投影和读写模型的实践取舍。
- `mkopylec/project-manager`：从业务描述、通用语言和测试逐步完成战术建模的教学结构。
- `humank/ddd-practitioners-ref`：Event Storming、Bounded Context Canvas、Example Mapping 与 Specification by Example 的串联工作流。

最终核心案例可以根据代码证据质量在上述候选中调整，但不得扩展到非 JVM 样例。

### 扩展资料

收录 Awesome DDD 中其余有价值的 JVM Sample Projects，并从 Blogs 与 Community Resources 中选择能补充以下主题的资料：上下文映射、聚合设计、DDD 启发式、Event Storming、遗留系统采用策略、活文档与架构测试。扩展资料只做分类简评，不对所有链接等量展开。

### 排除项

- 不分析 Go、.NET、JavaScript/TypeScript、PHP 等语言的样例项目。
- 不把框架或库的使用方式直接提升为 DDD 原则。
- 不复制大段受版权保护的文档或源码；代码仅保留支撑结论所需的短片段，并链接原文件。
- 不在本阶段修改现有技能内容或项目模板。

## 信息筛选与证据规则

每个核心项目从六个维度分析：

1. 业务问题与通用语言是否明确。
2. 限界上下文、聚合和不变式是否有业务证据。
3. Application、Domain、端口与适配器的职责是否清楚。
4. 事务、领域事件、集成事件、CQRS 或事件溯源采用了什么一致性策略。
5. 单元测试、用例测试、集成测试或 ArchUnit 是否保护设计。
6. 哪些实践可迁移到当前技能，哪些只是该项目的条件化选择。

重要结论必须附至少一种证据：原仓库 README、官方项目文档、代表性源码文件、测试文件或被 Awesome DDD 直接引用的文章。报告记录访问日期，并标注明显陈旧、归档、教学性质或不能视为生产基线的项目。

## 页面架构

### `index.html`

作为唯一主入口，展示调研摘要、核心结论、推荐阅读路线和五个主题导航。首页突出“从业务发现到工程守护”的主线，不堆叠完整项目详情。

### `projects.html`

展示 6–8 个核心案例。每个案例包含：定位、值得阅读的原因、架构与模型要点、代表性代码或目录、局限、可迁移结论、原始链接。使用折叠区域容纳代码，避免首屏过长。

### `patterns.html`

按实践主题横向比较项目，而非重复逐项目介绍。主题包括：

- Event Storming 到代码的知识流转
- 限界上下文与模块化单体
- 聚合、不变式与类型表达
- 端口、适配器与依赖方向
- CQRS 的渐进采用
- 事件、一致性、顺序与幂等
- 持久化隔离和查询模型
- 测试金字塔、示例驱动测试与 ArchUnit

每个主题明确“稳定原则、团队默认值、条件化方案、示例”四种规则级别。

### `resources.html`

提供其余 JVM 项目、Blogs 与 Community Resources 的分类索引。每条包含一句用途说明、主题标签、阅读优先级和外部链接。该页追求检索效率，不进行长篇分析。

### `skill-gaps.html`

对照当前三个技能输出差距矩阵：现有覆盖、外部证据、缺口、建议吸收项、预期落点、风险或不采用原因。建议只作为后续讨论输入，不直接改动技能。重点审视：

- 建模前的协作式发现与示例映射是否充分。
- 上下文映射和上下文集成模式是否缺失。
- CQRS 是否应表达为渐进决策而非固定架构。
- 事件顺序、幂等、投影恢复和事件演进是否需要补充。
- ArchUnit、模块边界测试和场景测试是否能成为生成器守护栏。
- 当前六模块模板是否把单一结构误当作所有上下文的统一答案。

### 公共资源

- `assets/styles.css`：统一排版、颜色、响应式布局、代码块和打印样式。
- `assets/app.js`：仅负责筛选、折叠、当前导航和轻量交互；无第三方依赖。

所有页面通过相对链接互相导航，直接双击 `index.html` 即可使用，不需要本地服务器或构建工具。

## 视觉与交互方向

采用“架构研究手册”风格：偏纸张与工程蓝图的暖灰底色，深墨色正文，以橙红和蓝绿色分别表示业务发现与工程实现。使用具有编辑感的标题字体组合和清晰的代码等宽字体，不使用常见紫色渐变或通用仪表盘卡片风格。

布局保持高信息密度但不过度拥挤。桌面端使用侧边目录和非对称内容网格，移动端退化为单列。交互只服务于阅读：主题过滤、代码展开、跳转锚点和打印友好，不加入无关动画。

## 文件体量与性能

- 每个 HTML 只承载一个主题，目标控制在约 100 KB 以内。
- 公共样式和脚本只加载一次，避免页面间复制。
- 不引入前端框架、外部字体运行时或 CDN 依赖。
- 不嵌入大图片；必要图示优先用 HTML/CSS 或小型内联 SVG 表达。
- 外部链接使用新窗口打开，并保留清晰的来源域名。

## 错误与不确定性处理

- 链接失效时保留项目名、原 URL、检查日期和失效标记，不用未经核实的镜像替代。
- README 与源码表达冲突时，以源码和测试为优先证据，并在报告中说明差异。
- 无法确认的架构意图标为“推断”，不写成项目作者的明确结论。
- 过时技术写法与仍然有效的设计思想分开评价。
- 所有对当前技能的建议都标注置信度和证据来源，避免把单个案例泛化为普遍规则。

## 验证与验收

完成后执行以下验证：

1. 检查所有本地页面、样式和脚本引用不存在断链。
2. 抽查全部外部核心来源链接可访问，并记录访问日期。
3. 检查每个核心案例都覆盖六个分析维度或明确说明不适用项。
4. 检查代码片段能在原仓库找到对应来源，且长度仅满足评论需要。
5. 在桌面和窄屏尺寸下打开主页面与至少两个子页面，检查导航、折叠和筛选。
6. 检查浏览器控制台无错误，页面无需服务器即可工作。
7. 检查 `skill-gaps.html` 的每条建议都有证据、落点和不采用风险。

验收标准是：读者可以从 `index.html` 在五分钟内理解核心结论；可以继续定位到具体项目和代码证据；并能从差距页形成一份有依据的技能优化讨论清单。
