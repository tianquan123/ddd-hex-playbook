# DDD 技能决策质量优化设计

**日期：** 2026-07-15  
**状态：** 已批准，待实施计划  
**范围：** `ddd-modeling`、`java-spring-hex-playbook`、仓库级技能评测与元数据

## 1. 背景

当前仓库用三项技能串联从业务场景到 Java/Spring 工程的过程：

```text
ddd-modeling
  → java-spring-hex-playbook
  → ddd-project-create
```

现有技能已经明确了从场景和不变式推导模型、按消费者放置端口、区分领域事件与集成事件等原则；`ddd-project-create` 也有确定性生成器、Python 契约测试和 Maven 构建验证。

本次 JVM DDD 案例调研暴露出的主要差距不在模板数量，而在模板之前的决策链：

- 业务发现被压缩为字段收集，没有稳定保留事件时间线、策略、热点、示例与反例；
- `bounded-contexts.md` 已包含关系模式，但主工作流和输出契约没有要求形成可交接的 Context Map；
- 建模结论与 Java/Spring 工程决策之间缺少固定交接格式；
- CQRS 缺少从轻量职责分离到独立存储、Event Sourcing 的渐进判断；
- 事件规则分散在事务和消息参考中，缺少按事件语义裁剪的统一审查入口；
- 两项决策技能没有行为评测，内容变化无法证明决策质量提高；
- `ddd-modeling` 和 `java-spring-hex-playbook` 的 UI 中文简介存在编码异常风险。

因此第一轮优化选择“决策质量优先”，暂时冻结项目生成模板。

## 2. 目标

第一轮优化必须达到以下结果：

1. `ddd-modeling` 能根据问题完整度选择快速评审或发现建模流程。
2. 建模结果形成可复核、可交接的结构化决策包。
3. 上下文映射只覆盖当前问题，并明确事实权威、上下游和翻译责任。
4. `java-spring-hex-playbook` 能按限界上下文选择局部架构。
5. CQRS 建议采用最小充分复杂度，并说明升级证据和停止位置。
6. 事件审查能区分本地领域事件、可靠集成事件和 Event Sourcing。
7. 静态契约、路由契约和行为案例能够验证技能行为。
8. 简单问题继续获得简洁答案，不被完整发现流程拖累。

## 3. 非目标

第一轮明确不做：

- 不新增 `ddd-discovery`、`ddd-context-mapping` 或总流程编排技能；
- 不修改六模块工程模板；
- 不引入 ArchUnit 或新的模板依赖；
- 不生成 CRUD、CQRS、异步投影或 Event Sourcing 代码模板；
- 不自动创建或修改 `CONTEXT.md`、Context Map、ADR；
- 不把 EventStorming、CQRS、Outbox 或 Event Sourcing 规定为必经步骤；
- 不把六模块解释为每个限界上下文都必须复制的内部结构。

## 4. 方案选择

采用“增强现有两项决策技能”的方案，不拆分新技能，也不增加编排技能。

选择理由：

- 现有触发边界清楚：业务模型归 `ddd-modeling`，Java/Spring 落位归 `java-spring-hex-playbook`；
- 新增专项技能会产生触发重叠和交接成本；
- 在行为评测成熟前增加总编排器，会复制规则并放大上下文占用；
- 现有 reference 结构适合通过一级路由渐进披露新增知识。

## 5. 技能协作架构

```text
用户问题
  ↓
ddd-modeling
  ├─ 快速评审轨
  └─ 发现建模轨
  ↓
建模决策包
  ├─ 场景与证据
  ├─ 不变式与一致性
  ├─ 上下文小地图
  ├─ 候选模型与反例
  └─ 工程交接契约
  ↓
java-spring-hex-playbook
  ├─ 局部架构选择
  ├─ 职责、端口与事务
  ├─ CQRS 决策梯度
  └─ 三档事件审查
  ↓
可执行的工程决策与验证方式
```

三个技能的职责保持不变：

- `ddd-modeling` 负责业务语言、规则、边界与模型候选，不规定 Java 包或框架实现；
- `java-spring-hex-playbook` 接收已验证的建模证据，决定职责、依赖、端口、事务和技术落位；
- `ddd-project-create` 继续生成团队当前批准的六模块基线，不参与第一轮行为扩展。

## 6. `ddd-modeling` 设计

### 6.1 自适应双轨

技能根据输入选择流程，而不是要求用户预先指定深度。

**快速评审轨**适用于用户已经提供明确业务场景或现有模型：

1. 读取现有模型、代码、`CONTEXT.md` 和 ADR；
2. 提取已有场景、不变式和一致性边界；
3. 用反例挑战实体、聚合和上下文划分；
4. 输出保留、调整和排除的设计。

**发现建模轨**适用于业务事实不足或问题涉及上下文划分：

1. 建立参与者、命令、事件、策略和外部系统时间线；
2. 标记歧义语言、争议点和缺失事实；
3. 使用具体示例和反例澄清规则；
4. 区分即时一致、延迟一致和补偿规则；
5. 提出上下文与聚合候选；
6. 用跨边界失败场景验证候选；
7. 输出 Context Map 和工程交接契约。

现有 direct-answer exception 保留。定义题不进入工作坊式流程。

### 6.2 建模决策包

统一输出七部分：

1. 场景范围与业务时间边界；
2. 证据、示例、反例与热点；
3. 通用语言及歧义；
4. 不变式和一致性要求；
5. 上下文、聚合及事件候选；
6. 排除方案与未验证假设；
7. 工程交接契约。

输出默认保留在对话中。只有用户明确要求持久化时，才修改项目文档。

### 6.3 小型 Context Map

上下文映射是候选上下文稳定后的可选阶段，不拆为独立技能。

默认产物包括：

- 一张只覆盖当前问题的小地图；
- 一张关系表；
- 一份热点和待验证假设列表。

每个上下文描述业务目标、本地语言、关键事实和事实权威。每条关系描述上游、下游、契约定义方、翻译责任、兼容成本和失败影响。先描述关系事实，再标注 Partnership、Shared Kernel、Customer/Supplier、Conformist、ACL、Open Host Service、Published Language 或 Separate Ways 等模式。

### 6.4 工程交接契约

交接信息至少包含：

- 当前限界上下文及业务目标；
- 命令、用例和成功/失败结果；
- 聚合及必须即时成立的不变式；
- 允许延迟或需要补偿的规则；
- Domain Event 与跨上下文事实；
- 上下游、事实权威和翻译责任；
- 查询需求与一致性窗口；
- 未验证假设和热点；
- 明确禁止固化的候选设计。

## 7. `java-spring-hex-playbook` 设计

### 7.1 交接检查

工程技能先检查建模证据是否足以支持决策。缺少会改变职责、事务或一致性方案的事实时，退回 `ddd-modeling` 并一次提出一个关键问题，不自行补造业务事实。

### 7.2 按上下文选择局部架构

同一部署单元允许不同上下文采用不同策略：

| 局部策略 | 适用信号 | 设计重点 |
| --- | --- | --- |
| 简单 CRUD / Transaction Script | 规则少、生命周期简单 | Application 直接编排数据访问，不强造聚合和事件 |
| 富领域模型 / Hexagonal | 存在明确不变式和行为生命周期 | 聚合保护一致性，Application 编排，端口由消费者拥有 |
| 独立查询 / CQRS Read Side | 列表、报表或复杂查询与写模型明显不同 | 按查询用例组织读模型，不为读取重建聚合 |

六模块继续表达编译、发布和装配边界，不规定每个上下文内部必须复制六层。

### 7.3 CQRS 最小充分复杂度

CQRS 使用六级决策梯度：

1. 同一模型中分离命令与查询方法；
2. 分离 Command/Query 接口；
3. 写模型与读模型分离，但共用数据库；
4. 异步投影读模型；
5. 独立读写存储；
6. Event Sourcing。

技能必须说明：

- 当前建议停在哪一级；
- 触发升级的具体证据；
- 查询压力和读模型差异；
- 可接受的一致性窗口；
- 投影失败后的恢复方式；
- 新增运维成本；
- 为什么不选择更高一级。

这不是成熟度阶梯。大多数系统停在前两级是正常结果，Event Sourcing 不是默认终点。

### 7.4 三档事件审查

**本地 Domain Event**检查事实名称、事实所有者、聚合来源、技术无关性和事务内收集方式。

**可靠 Integration Event**进一步检查外部契约与版本、Outbox 原子写入、分区键、顺序、消费者幂等、重试、DLQ、未知结果和契约兼容。

**Event Sourcing**在前述基础上增加事件版本、乐观并发、快照、回放、投影重建、检查点、schema 演进和历史迁移。

Outbox 是可靠投递链路的一部分，不是事件正确性设计的终点。Event Sourcing 的要求不得无条件施加给普通领域事件。

### 7.5 统一输出

工程决策输出包括：

- 推荐方案和适用上下文；
- 规则级别；
- 支撑方案的业务证据；
- 职责和端口所有者；
- 依赖、事务和一致性影响；
- CQRS 当前级别及升级触发条件；
- 事件档位及正确性清单；
- 未选择方案及原因；
- 可执行验证；
- 需要退回建模阶段的问题。

## 9. 评测设计

### 9.1 静态契约测试

使用 Python 标准库验证：

- 三个 `SKILL.md` frontmatter 可解析，技能名与目录一致；
- 所有直接 reference 路径存在；
- 不引入两级以上 reference 跳转；
- `agents/openai.yaml` 可按 UTF-8 读取，且 UI 描述与技能用途一致；
- 输出契约和停止条件存在；
- 生成器原有测试继续通过。

### 9.2 路由契约测试

| 问题 | 必须路由 |
| --- | --- |
| 业务发现、热点、示例 | `discovery-evidence.md` |
| 上下文关系、事实权威 | `context-mapping.md` |
| 模块与局部架构 | `module-strategies.md` |
| CQRS 升级判断 | `cqrs-decision-ladder.md` |
| 事务、Outbox、事件溯源 | `transactions-and-events.md` |

### 9.3 行为评测

首批十个案例覆盖：

1. 根据数据库表推导聚合——必须拒绝；
2. “账户”在身份和结算中的歧义——必须提出上下文候选；
3. Order 与 Payment 是否同聚合——必须从即时一致性举证；
4. 简单商品目录——不应强造富领域模型；
5. 复杂订单生命周期——应选择富领域策略；
6. 普通分页查询——不应直接拆库或异步投影；
7. 复杂运营报表——应明确 CQRS 当前级别和一致性窗口；
8. 本地 Domain Event——不应自动要求 Outbox；
9. 跨上下文 Integration Event——必须覆盖幂等、顺序和失败；
10. Event Sourcing——必须覆盖回放、版本、投影恢复和 schema 演进。

每个案例记录用户提示、必须出现的决策行为、禁止行为、必须追问的问题、允许的替代方案和评分规则。评分按行为而非逐字匹配。

每次行为评测使用全新的 Codex 会话运行。运行时只提供目标技能、案例提示和案例要求的原始项目材料，不提供评分规则、预期答案或本次优化意图，避免上下文泄漏。每个 run 保存：

- 技能所在提交、运行日期和模型信息；
- 每个案例的原始响应；
- 按 rubric 得到的分项分数和关键禁行项结果；
- 与基线 run 的变化摘要。

如果当前环境不能自动创建独立会话，允许手工逐案运行，但仍使用相同的输入隔离和结果格式。行为评测不采用固定文本快照，也不把关键词命中等同于决策正确。

### 9.4 发布门槛

- 静态和路由测试 100% 通过；
- 行为评测总分至少 90%；
- 任一关键禁行项失败时不得发布；
- 生成器原有 Python 测试与模板 Maven 构建不得回归；
- 简单定义题和已有充分证据的评审题不得被强制拉入完整发现流程；
- 两项决策技能保持渐进披露，详细规则只存在于一级 reference。

## 10. 失败与边界处理

- 缺少会改变模型的业务事实时，一次只追问一个问题；
- 项目代码或 ADR 与技能默认值冲突时，以项目事实为准并指出影响；
- 用户只要求分析或评审时，不自动修改文件；
- 不因看到事件、查询或六边形架构就自动引入 Outbox、CQRS 或六模块；
- 模式名称不能代替关系事实，技术结构不能充当领域证据；
- 无法证明复杂方案必要时，选择更低复杂度方案并给出未来升级触发条件。
