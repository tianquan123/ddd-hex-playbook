# Entities and Value Objects

## Decision

- 当业务需要追踪“同一个事物”在变化中的连续性时，使用 Entity。
- 当对象按属性值可互换、变化通过整体替换表达时，使用 Value Object。
- 身份、生命周期和业务语言决定类型；可变字段、数据库主键和 ORM 映射不能单独决定。

## Questions to ask

1. 两个属性完全相同的实例，在业务上是否仍可能是不同对象？
2. 用一个等值实例替换它，业务含义是否改变？
3. 业务是否需要追踪它从过去到未来仍是“同一个”？
4. 它能否独立存在，还是只表达另一个对象的属性或度量？

## Equality and behavior

- Entity 的相等性基于稳定身份，不把全部可变字段加入判等。
- Value Object 的相等性基于完整语义值，并保持不可变。
- 行为放在最能用领域语言表达并守护规则的对象上，不因 DTO 形状创建贫血模型。

## Creation and reconstruction

创建策略和生命周期不变式必须分开：

- 创建策略只约束新实例，例如初始状态必须为 `PENDING`。
- 生命周期不变式在每个有效状态中都成立，重建也不能绕过。
- 重建可以跳过 ID 生成、创建事件和创建期副作用。
- 重建不能静默产生领域层自己不信任的对象。

历史数据与当前模型不兼容时，选择数据迁移、隔离、版本化重建，或显式建模 `Legacy/Unknown` 状态；不要用“绕过全部校验”掩盖冲突。

## Common mistakes

- 有主键就判为 Entity。
- 不可变类就判为 Value Object。
- 用 Lombok、record 或类后缀代替领域判断。
- 让 `reconstitute` 绕过所有不变式。
- 把 ORM 持久化便利性当作模型边界。
