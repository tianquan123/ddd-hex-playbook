# 评审结论

不建议共享一个“标准 Account”领域模型。这里的 `Account` 是同名异义：身份上下文中的“登录账号”承载登录主体、凭证和停用生命周期；结算上下文中的“资金账户”承载余额以及资金冻结、解冻规则。二者的业务目标、规则、事实权威和生命周期不同。两个团队独立发布也使 Shared Kernel 的协同变更与发布耦合成本不合适。

可以共享的只应是经过验证的最小、版本化集成契约，而不是实体、状态枚举或完整生命周期。当前事实尚未证明“登录账号停用”必须触发任何结算动作，因此不能先发明这条依赖。

## 小型 Context Map

```text
[身份上下文]
  本地概念：登录账号
  权威事实：登录、凭证、账号停用
        |
        | 目前没有已验证的跨上下文业务事实
        | 若评审确认停用会影响资金操作：仅发布版本化的最小身份状态契约
        v
  [结算侧翻译边界（候选）]
        |
        v
[结算上下文]
  本地概念：资金账户
  权威事实：余额、资金冻结、资金解冻
```

当前关系应暂记为 **Separate Ways（候选）**：各自维护本地模型，不共享标准实体。只有确认存在跨边界规则后，才将关系调整为“身份上下文提供版本化契约，结算上下文负责翻译”的集成关系；关系模式标签届时再定。

## 热点列表

| 热点 | 影响的模型决定 | 事实负责人 | 评审验证方式 |
| --- | --- | --- | --- |
| 一个登录账号是否必然对应一个资金账户，还是一对多、多对一或可不存在 | 是否需要跨上下文标识映射及映射归属 | 两团队的业务负责人 | 用开户、合并身份、迁移资金账户、无资金账户用户等实例和反例确认 |
| 登录账号停用是否必须阻止结算操作 | 是否需要身份 → 结算的事实流，以及结算侧本地状态/策略 | 身份与结算业务负责人共同确认，结算负责人定义资金规则 | 分别评审停用后查询余额、入账、出账、冻结和解冻场景 |
| “停用”是否等同于“冻结” | 是否错误合并两个生命周期 | 身份负责人解释停用语义；结算负责人解释冻结语义 | 对比触发者、原因、恢复条件、审计要求；寻找停用但不冻结、冻结但不停用的反例 |
| 若停用影响资金操作，允许多大传播延迟 | 即时、延迟或补偿的一致性选择 | 两团队及风险/合规负责人 | 明确消息延迟、身份系统不可用和旧状态下继续处理的可接受结果 |
| 跨边界标识使用什么稳定键，是否允许重建或合并 | 契约字段、映射生命周期和幂等处理 | 两团队的数据/领域负责人 | 演练账号合并、重新注册、资金账户迁移及重复事件 |
| 契约变更和故障由谁负责 | 独立发布能否维持，以及是否需要版本化 Published Language/ACL | 两团队技术负责人 | 定义兼容窗口，并演练重复、乱序、不兼容、超时和未知结果 |

在这些热点被验证前，两边应明确使用“登录账号”和“资金账户”，避免裸用 `Account`；不共享实体类、状态枚举或数据库表作为领域契约。
