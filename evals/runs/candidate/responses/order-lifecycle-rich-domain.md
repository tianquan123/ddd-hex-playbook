推荐采用 **富领域模型 / Hexagonal** 作为订单上下文的局部架构。证据是订单存在明确生命周期和必须即时守护的不变式，而不是简单字段 CRUD。

**Domain（架构不变量）**：以 `Order` 作为聚合根，把状态转换和规则放进具名行为，例如 `changeQuantity`、`confirm`、`cancel`、`ship`，不要暴露公共 setter 让 Application 拼状态。聚合必须保证：确认前可以调整数量；确认后数量不可修改；发货后不可取消。订单项数量若只能随订单一起修改并参与“确认后冻结”的规则，应由 `Order` 通过聚合内实体管理。上述规则自然属于订单，不需要另建 DomainService。Domain 不依赖 Spring、HTTP、Job 或持久化类型。

**Application（架构不变量）**：分别组织确认、取消、发货、调整数量等用例。每个用例加载 `Order`，调用对应领域行为，在一个本地事务中保存该聚合，并把领域拒绝作为应用失败返回；Application 不重复判断订单状态。`OrderRepository` 等出站端口由实际消费它的 Application 侧拥有，Infra 实现。HTTP 与定时任务可能并发操作同一订单时，应使用版本号/乐观锁保护，并在冲突后重新加载订单、重新执行领域规则，而不是盲重试 SQL。

**Trigger（架构不变量）**：HTTP 入口只负责路由、请求转换和 HTTP 错误映射；定时关闭 Job 只负责调度参数、分片/并发策略并调用同一个 Application 关闭或取消用例。两种入口都不能各自复制“确认后冻结数量”“发货后不能取消”等规则。跨入口的业务幂等若有要求，应由 Application 定义语义和端口，不能只在 HTTP 或 Job 内去重。

**事务与一致性**：一次命令只修改一个 `Order` 聚合，状态转换和数量规则在命令返回前原子成立。当前事实没有跨上下文可靠通知需求，因此不默认引入 Domain Event、Outbox、消息总线或 Event Sourcing；只有确认、取消或发货成功后确有跨边界后续动作时，才按可靠性要求补充事件方案。

不选择 Transaction Script，因为它会把持续增长的状态判断推到 Application、Controller 或 Job；也不按 HTTP 与 Job 拆两套业务实现，因为入口协议不是领域边界。现有 Maven 模块只承载上述职责，不需要为每个用例新增模块。

验证应包括：不启动 Spring 的纯 Domain 单元测试，覆盖“确认前改数量成功、确认后改数量失败、发货后取消失败”以及合法状态转换；Application 测试验证加载—调用行为—保存及回滚；HTTP/Job 切片测试只验证转换和触发；并发集成测试验证 HTTP 与 Job 竞争时只有符合最新状态的转换提交。执行项目现有的 `mvn test`、`mvn verify` 和依赖守卫。仍需业务确认重复确认/取消/发货是否幂等，以及定时关闭具体允许作用于哪些状态；这些答案影响失败语义，但不改变上述职责落位。
