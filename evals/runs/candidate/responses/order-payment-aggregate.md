# Order 与 Payment 聚合建模决策包

## 结论

**Order 与 Payment 不应放在同一个聚合。** 当前证据支持把它们建模为两个独立聚合：`Order` 负责订单生命周期与取消规则，`Payment` 负责支付尝试、支付结果及支付侧补偿；两者通过身份引用和业务事件协作。

理由不是“需要异步”或“可能拆成两个服务”，而是二者不存在已知的跨对象即时不变式，且已经呈现不同的业务时间边界、失败路径和补偿规则：订单确认后可以在未支付状态等待；支付结果可能晚到或失败；支付可被独立补偿；订单取消与退款又受不同规则控制。把二者合并会把本可延迟或补偿的一致性误升格为单事务一致性，并耦合两个独立生命周期。

限界上下文层面，当前可提出“订单上下文”和“支付上下文”两个**候选**，但仅凭现有事实还不足以把组织、部署或数据库边界写死；需要进一步确认术语、规则权威和团队责任。

## 1. 场景范围与业务时间边界

本决策只覆盖“订单确认后等待支付，随后接收支付成功、失败、晚到或补偿结果，以及订单取消与退款之间的协作”。

关键业务时点：

1. 订单被确认，进入可等待支付的状态。
2. 支付被发起并形成独立的支付处理过程。
3. 支付可能成功、失败，也可能在订单状态已经变化后才返回结果。
4. 订单可能按订单取消规则被取消。
5. 已发生的资金事实可能按退款或支付补偿规则处理。

这些时点之间存在可观察的时间间隔，因此“订单已确认但尚未支付”是合法状态，而非必须在一次命令中消除的临时错误。

## 2. 证据、示例、反例与热点

### 已给定事实

- 订单确认后可以等待支付。
- 支付可能晚到、失败或被独立补偿。
- 订单取消和退款由不同业务规则控制。

### 支持拆分的示例

- `Order` 完成确认后保持“等待支付”，即使此时不存在成功的 `Payment`，订单自身仍然有效。
- 支付失败只改变支付尝试的结果，不必原子回滚“订单曾被确认”这一业务事实；后续如何处理订单由订单策略决定。
- 支付晚到时，需要依据订单当前状态决定接纳支付结果还是触发后续补偿。这是跨聚合流程判断，不是把历史上两个动作压入一个事务。
- 订单取消与退款分别由各自规则判断；取消成功不等于退款必然已完成，退款完成也不应反向改写订单取消规则。

### 反例挑战

- 若业务方能证明存在规则“订单确认命令返回前，支付必须成功且两者要么同时成功要么同时失败”，则当前边界需要重审；但它与“确认后可以等待支付”的已知事实冲突。
- 若某类订单确有“支付到账与订单生效必须不可分割”的特殊即时不变式，应把它作为独立场景举证，而不能据此扩大所有订单的聚合。
- 即便 Order 与 Payment 当前共享数据库、外键或事务，也不能据此合并聚合；这些是实现现状，不是业务证据。

### 热点

- 订单取消后晚到的支付，究竟应被拒绝、记为已收款后退款，还是进入人工处理？
- 一张订单是否允许多次支付尝试、部分支付或多笔退款？
- “支付补偿”是否等同于退款，还是支付渠道的冲正/撤销？两者的术语和规则权威可能不同。
- 支付成功后订单状态最迟多久必须可见？超时后由谁处理？
- 退款的发起资格由订单事实、支付事实还是独立售后事实共同决定？

## 3. 通用语言与歧义

| 术语 | 本决策中的含义 | 需要避免的歧义 |
| --- | --- | --- |
| 订单确认 | 订单侧已经接受订单并允许进入等待支付阶段 | 不等同于支付成功 |
| 支付 | 具有自身身份和生命周期的资金处理过程或尝试 | 不等同于订单的一个可随意内嵌字段 |
| 支付成功 | 支付侧确认资金结果的业务事实 | 不自动等同于订单已完成 |
| 订单取消 | 依据订单规则终止或改变订单履约资格 | 不等同于退款 |
| 退款 | 依据退款规则返还已发生的资金 | 不等同于取消订单 |
| 支付补偿 | 对失败、晚到或异常支付结果采取的反向或修正动作 | 是否等同于退款尚待业务确认 |

## 4. 不变式与一致性要求

### Order 聚合候选守护的即时不变式

- 订单状态变更必须符合订单生命周期和取消规则。
- 订单确认、取消等命令返回前，Order 自身状态必须合法。
- Order 只能记录所需的 `PaymentId`、支付状态摘要或已接收的支付事实，不直接修改 Payment 内部状态。

### Payment 聚合候选守护的即时不变式

- 支付尝试及其成功、失败、补偿状态变更必须符合支付生命周期。
- 支付结果只能按照支付侧规则推进，重复或冲突结果不能破坏 Payment 自身状态。
- Payment 通过 `OrderId` 关联业务目的，不直接修改 Order 内部状态。

### 跨聚合一致性

- “支付结果已产生”到“订单已反映该结果”允许延迟；具体窗口尚待确认。
- 支付失败、晚到及独立补偿均按流程推进，不要求 Order 与 Payment 原子提交。
- 订单取消与退款分别判定；二者若需联动，应通过显式流程协调和补偿，而不是共享一个聚合根。
- 目前没有证据支持跨 Order 与 Payment 的即时强一致。若未来提出该要求，必须指出具名业务规则、容忍窗口以及补偿不可接受的理由。

## 5. 上下文、聚合与事件候选

### 小地图

```text
[订单候选上下文 / Order 权威]
    -- OrderConfirmed、OrderCancelled 等最小事实 -->
[翻译与流程协调边界]
    --> [支付候选上下文 / Payment 权威]

[支付候选上下文 / Payment 权威]
    -- PaymentSucceeded、PaymentFailed、PaymentCompensated 等最小事实 -->
[翻译与流程协调边界]
    --> [订单候选上下文 / Order]
```

### 关系表

| 上游 | 下游 | 事实权威 | 传播内容 | 翻译责任 | 一致性 | 失败责任与热点 |
| --- | --- | --- | --- | --- | --- | --- |
| 订单候选上下文 | 支付候选上下文 | Order 对确认、取消等订单事实有权威 | `OrderId`、订单确认或取消事实及必要业务属性 | 支付侧将订单事实翻译为本地支付资格/处理输入 | 延迟 | 重复、乱序、取消晚于支付发起时如何处理待确认 |
| 支付候选上下文 | 订单候选上下文 | Payment 对支付成功、失败、补偿等资金处理事实有权威 | `PaymentId`、`OrderId`、结果、发生时间及必要金额/币种信息 | 订单侧将支付事实翻译为本地订单状态推进输入 | 延迟或补偿 | 晚到、重复、未知结果及订单已取消时的责任待确认 |

关系模式暂不预选。只有契约定义方、兼容责任和双方影响力明确后，才决定是否采用 Published Language、Customer/Supplier 或其他模式。

### 领域事件候选

- Order 产生：`OrderConfirmed`、`OrderCancelled`。
- Payment 产生：`PaymentSucceeded`、`PaymentFailed`、`PaymentCompensated`。
- 是否存在 `RefundRequested`、`RefundSucceeded` 等事件，取决于退款是否为独立模型及其事实权威，当前不能写死。

事件名称表达已发生的业务事实；事件只携带当时已确定的最小业务数据。消息中间件、Outbox、重试和投递语义不属于本建模决定。

## 6. 被排除方案与未验证假设

### 被排除方案

1. **Order 内嵌并直接管理 Payment。** 排除证据：订单可等待支付，支付可晚到、失败或独立补偿，说明 Payment 不必与 Order 在同一原子变更中保持合法。
2. **取消订单等同于退款。** 排除证据：两者明确由不同业务规则控制。
3. **支付失败自动回滚订单确认。** 排除证据：确认后的等待支付是合法状态；失败后的订单策略没有给定，不能补造。
4. **因共享数据库或调用链而合并聚合。** 排除理由：持久化和调用方式不是聚合边界证据。
5. **现在就固定两个微服务或两个部署单元。** 排除理由：聚合和候选上下文的结论不自动决定部署拓扑。

### 未验证假设

- 暂假设订单侧拥有订单状态与取消规则，支付侧拥有支付结果与补偿规则；需由相应业务负责人确认。
- 暂假设跨边界重复、延迟和乱序均可能发生，具体容忍窗口及人工兜底尚未确定。
- 暂假设退款可能需要独立职责，但现有事实不足以确认它是 Payment 内行为、独立聚合还是独立上下文。

## 工程交接契约

### 1. 当前上下文与业务目标

- 当前候选为订单上下文与支付上下文；目标是在允许订单等待支付的前提下，正确处理支付成功、失败、晚到、取消及补偿。
- 已确认聚合边界为独立的 `Order` 与 `Payment`；上下文、团队和部署边界仍为候选，不得自动等同。

### 2. 命令、用例与结果

- `ConfirmOrder`：成功结果是订单进入合法的已确认/等待支付阶段；失败结果是订单规则拒绝确认。
- `InitiatePayment`：成功结果是产生可追踪的支付过程；失败结果由支付规则表达，不反向抹除订单确认事实。
- `RecordPaymentResult`：接收成功或失败结果；重复、冲突、晚到结果必须有可判定结果，具体规则待业务确认。
- `CancelOrder`：按订单取消规则成功或被拒绝；不得隐含“退款已经完成”。
- 支付补偿/退款相关命令的命名、资格与结果需在术语负责人确认后固化。

### 3. 聚合与即时不变式

- `Order` 聚合：原子守护订单自身状态迁移、确认和取消规则；仅持有支付身份或所需事实摘要。
- `Payment` 聚合：原子守护支付尝试及成功、失败、补偿状态迁移；持有 `OrderId`，不直接修改 Order。
- 默认单次事务只修改一个聚合。任何多聚合原子事务要求必须回到建模层举证。

### 4. 延迟与补偿

- 支付结果传播到订单允许延迟；最大一致性窗口待业务负责人确认。
- 支付失败、晚到和独立补偿由显式流程协调；具体使用 Saga、流程管理器或其他机制由工程设计决定。
- 取消与退款不可合并为一个原子动作；需要明确各种顺序下的补偿责任和人工兜底。

### 5. 事件与跨上下文事实

- 本地域事件候选：`OrderConfirmed`、`OrderCancelled`、`PaymentSucceeded`、`PaymentFailed`、`PaymentCompensated`。
- 跨上下文契约只传播最小稳定事实，包括业务标识、结果、发生时间以及经确认必要的金额/币种；不得发送完整可变聚合快照。
- Domain Event 与对外集成消息应概念分离；发布时机、Outbox、MQ 和投递保证留给 Java/Spring 设计。

### 6. 关系、权威与翻译

- Order 是订单确认、状态和取消事实的候选权威；Payment 是支付结果和支付补偿事实的候选权威。
- 下游各自负责把外部事实翻译成本地命令或状态推进输入，不共享可直接修改的领域对象。
- 契约定义方、版本兼容策略、重试/死信/人工处理责任尚待双方业务与架构负责人确认。

### 7. 查询与一致性窗口

- 需要面向订单查询展示订单状态、最近已知支付状态及可能的补偿/退款进度；可采用组合读模型或投影，不因此合并写聚合。
- 读模型必须能表达“订单已确认但支付待定”“订单已取消但晚到支付待处理”等中间状态。
- 允许陈旧的最长时间、刷新提示和强制查询权威源的场景尚待产品/SLA 负责人确认。

### 8. 热点与未验证假设

- 订单取消后支付晚到的业务处理：由订单与支付业务负责人共同给出场景表。
- 多次/部分支付及多笔退款是否存在：由支付与财务规则负责人确认。
- 支付补偿与退款的语义边界及事实权威：由支付、财务/售后负责人确认。
- 一致性窗口、超时、重复和未知结果的处置：由业务负责人定义可接受结果，架构负责人据此选机制。
- 若任一答案要求跨聚合即时一致，必须退回建模并提供具名不变式及“补偿不可接受”的业务证据。

### 9. 禁止固化

- 不得把 Order 与 Payment 固化为同一聚合或共享可变对象图。
- 不得把候选上下文直接固化为两个微服务、两个数据库或固定 Java 包结构。
- 不得把同步调用、消息中间件、Outbox、exactly-once 或 Saga 实现选型写成领域事实。
- 不得把订单取消等同于退款，也不得在规则未确认前自动退款或自动回滚订单确认。
- 不得把事件候选的字段、版本和对外发布范围在业务语义未验证前冻结。
