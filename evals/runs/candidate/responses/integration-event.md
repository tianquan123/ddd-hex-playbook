# 结论与三档分类

本场景属于 **可靠 Integration Event**，应采用“订单业务状态与 Outbox 同一事务提交 + Outbox Relay 至少一次发布 + 履约端幂等消费”的端到端设计。

| 档位 | 判断 | 理由 |
| --- | --- | --- |
| 本地 Domain Event | 仅作为订单上下文内部事实使用 | 订单聚合成功确认后可以记录技术无关的 `OrderConfirmed` Domain Event，但本地事件不能满足独立部署上下文之间不可丢失的通知要求 |
| **可靠 Integration Event** | **选中** | 订单与履约独立部署，订单提交后即使代理不可用也必须最终发送；代理至少一次投递又要求履约端承受重复 |
| Event Sourcing | 不选 | 没有证据要求事件流成为订单聚合的事实来源，也没有审计、时态查询或从事件重建聚合的需求；可靠通知不能成为采用 Event Sourcing 的理由 |

`OrderConfirmed` 可以同时是内部事实名称和外部契约名称，但不能让同一个 Java 类未经转换同时承担 Domain Event 与长期 Integration Event 契约。以下设计属于跨进程可靠投递的架构不变量与本场景方案；具体 broker API、表结构和框架注解只是实现选择。

## 完整链路

```text
订单 Application 事务
  -> 执行订单聚合 confirm，聚合记录内部 OrderConfirmed Domain Event
  -> 保存订单
  -> 映射为版本化 OrderConfirmed Integration Event
  -> 写入 Outbox
  -> 同一数据库事务提交

Outbox Relay
  -> 并发安全地领取未发布记录
  -> 发布到消息代理
  -> 记录发布进度；失败退避重试

履约 Trigger/MQ Adapter
  -> 校验契约并调用履约 Application 用例
  -> 在同一履约数据库事务中：登记幂等键 + 创建履约任务
  -> 提交后 ack
```

不要在订单事务中直接双写数据库和 broker，也不要把远程发布纳入数据库事务。前者存在“订单已提交但消息未发出”的裂缝，后者无法把两个独立系统变成一个可靠原子事务。

## 可靠 Integration Event 检查项

### 1. 事实语义与所有者

- `OrderConfirmed` 表示“订单确认行为已经成功”，只能在订单聚合满足确认不变式并成功保存的同一业务事务中产生；命令收到、事务开始或消息准备发送都不等于该事实。
- 订单上下文拥有“订单已确认”事实；履约上下文拥有“履约任务是否创建及其状态”事实。
- 内部 Domain Event 保持技术无关，不携带 Spring、MQ、DO 或 broker 类型。

### 2. 外部契约

- 订单团队拥有并版本化 Integration Event schema；履约团队通过契约测试验证兼容性。破坏性变化发布新版本，约定兼容窗口和旧版本下线流程。
- 最小 envelope 至少包含：全局唯一且稳定的 `eventId`、事件类型、契约版本、`orderId`、事实发生时间；履约创建任务真正需要的业务字段必须逐项确认，不能直接序列化整个订单聚合。
- 若包含敏感字段，需明确最小披露、访问控制和保留期限。

### 3. 订单事务与 Outbox 原子性

- 订单状态更新与 Outbox 插入必须由订单 Application 用例控制，在同一数据库事务中提交。
- 事务回滚时二者都不存在；事务提交时即使 broker 停机，Outbox 仍保留可投递记录。
- Outbox 记录至少保存 `eventId`、类型、版本、payload、创建时间、发布状态/尝试次数；`eventId` 在重试中不得重新生成。

### 4. Relay、进度与未知结果

- Relay 需要并发安全领取、批量限制、退避重试、租约/超时恢复，避免多实例永久占有记录。
- broker 返回成功后、Outbox 标记发布前崩溃属于未知结果，Relay 必须允许再次发布；不能用“已调用 publish”推断只发送一次。
- 监控 Outbox 最老未发布时长、积压量、发布失败率和重试次数，并设置告警。已发布记录按审计与容量要求归档/清理。

### 5. 顺序与分区

- 若履约只关心同一订单内事件次序，则以 `orderId` 作为分区/顺序键，并只承诺每个订单的局部顺序；没有证据时不承诺全局顺序。
- 消费者仍要处理重复、延迟，以及可能的乱序；若前置事实缺失，应暂存/重试、查询权威源或进入可恢复异常流程，具体策略由业务时序要求决定。

### 6. 履约幂等与原子效果

- 至少一次投递意味着重复是正常情况。履约端使用 `(consumerName, eventId)` 作为通用幂等键；若业务规定一个订单只能有一个履约任务，再给 `orderId` 对应的任务建立唯一业务约束作为第二道防线。
- “幂等记录成功”和“创建履约任务”必须在履约侧同一数据库事务提交。先记幂等再在另一个事务建任务会丢效果，先建任务再单独记幂等会重复创建。
- 重复消息命中已成功记录时直接返回成功并 ack，不再次创建任务。并发重复由唯一约束收敛，并将冲突识别为已处理，而非无条件重试业务操作。
- 业务事务提交后 ack；提交成功但 ack 丢失会导致重投，仍由同一幂等机制安全处理。

### 7. 重试、DLQ 与恢复

- broker 暂不可用、网络超时等可恢复错误采用有上限的退避重试；契约无法解析、版本不支持、必填数据非法等不可恢复错误进入隔离/DLQ，避免毒消息阻塞分区。
- DLQ 必须可观测，并有负责人、诊断信息、修复后回放流程。回放仍使用原 `eventId`，继续经过幂等检查。
- 对“发布成功但确认未知”和“消费提交成功但 ack 未知”明确按可能成功处理，依靠重复投递与幂等收敛，不宣称 exactly-once。

## 职责与依赖

- 订单 Domain：确认订单并记录内部领域事实，不依赖消息设施。
- 订单 Application：控制订单与 Outbox 的事务，拥有写 Outbox/可靠发布所需的出站端口，并负责 Domain Event 到 Integration Event 的边界转换。
- 订单 Infra：实现订单持久化、Outbox 持久化与 Relay/broker 适配。
- 履约 Trigger MQ Adapter：承担反序列化、基础契约校验、ack/nack 和协议错误映射，不承载创建任务的业务规则。
- 履约 Application/Domain：执行幂等用例并创建履约任务；履约 Infra 实现 inbox/幂等记录和任务持久化。

## 必须通过的验证

1. 订单事务回滚：订单未确认，且不存在可投递 Outbox。
2. 订单事务提交、broker 停机：Outbox 保留；broker 恢复后最终发布。
3. Relay 发布成功但未标记即崩溃：重启后重复发布，履约只产生一个任务。
4. 履约事务提交但 ack 丢失：重投后仍只有一个任务。
5. 两条相同消息并发到达：幂等唯一约束与业务唯一约束均能收敛。
6. 验证延迟、重复、必要的局部乱序、不可兼容版本、重试耗尽、DLQ 修复回放。
7. Contract test 验证当前与兼容版本 schema；端到端监控验证从订单提交到履约处理的延迟、Outbox 积压和 DLQ 告警。

仍需两个团队明确的业务热点只有：履约创建所需的最小字段、同一订单是否需要严格事件顺序、不可恢复消息的人工责任人与最终处理时限。这些未知项不影响选用可靠 Integration Event + Outbox + 幂等消费的结论，但必须在投产契约中补齐。
