# 结论

不应为这个普通订单分页列表引入异步投影或独立读库。当前建议停在 **CQRS 第 2 级：分离 Command/Query 接口，但仍使用同一服务、同一数据库**。

分页列表由 Application 层拥有一个 `OrderQueryPort`，查询适配器直接用针对列表的 SQL 返回 `OrderSummary`/分页结果，不为读取重建订单聚合；写用例继续通过聚合 Repository。这里的“CQRS”只是职责和接口分离，不意味着消息总线、异步、独立服务或独立数据库。

**规则级别：**“选择满足证据的最小充分复杂度”是本次架构决策原则；第 2 级是基于当前事实的条件化方案。具体 SQL、分页方式和 DTO 字段属于实现选择。

## 六级决策梯度

| 级别 | 本场景判断 | 升级到该级所需的触发证据 |
| --- | --- | --- |
| 1. 同一模型内分离命令与查询方法 | 可以工作，但分页列表容易把页面字段、筛选条件带入聚合 Repository；因此推荐多走一步到第 2 级 | 若当前代码极小且查询、写模型完全同形，可暂留第 1 级；出现稳定的列表用例 DTO 或独立查询消费者时进入第 2 级 |
| **2. 分离 Command/Query 接口** | **当前级别。** `OrderQueryPort` 服务分页读取，写侧 Repository 服务聚合；同步同库 | 当前已具备分页查询这一独立读取职责，但没有更高等级所需的性能或一致性证据 |
| 3. 写模型与读模型分离、共用数据库 | 不选。查询只涉及订单表和少量筛选，当前数据库满足延迟目标，没有明显不同的查询组合、索引或读模型形状 | 可观察到复杂多表组合、聚合加载明显浪费、专用索引/物化读结构确有必要，且第 2 级 SQL 优化后仍不能满足延迟或吞吐目标 |
| 4. 异步投影读模型 | 不选。没有证据表明写事务无法承担同步维护，也没有业务接受陈旧读取的明确要求 | 第 3 级同步维护成为写事务瓶颈，并且业务明确接受一个量化的一致性窗口；同时团队能承担幂等、检查点、乱序/重复、重试、监控和可重建投影 |
| 5. 独立读写存储 | 不选。当前同库已满足延迟目标，没有读负载、存储形态、独立扩缩容或故障隔离需求 | 读流量造成持续连接数/CPU/IO 争用，或读侧需要不同存储形态、独立容量和故障隔离，且复制链路及一致性排障成本可接受 |
| 6. Event Sourcing | 不选。订单事件流没有“必须成为事实来源”的证据，分页查询更不是采用它的理由 | 业务要求以事件流作为权威事实，并对完整审计、时态查询或状态回放有明确价值，同时接受事件版本、重建、迁移和长期运维成本 |

## 为什么停在第 2 级

- 查询范围小：只查订单表并使用少量筛选，没有独立读模型的业务或技术证据。
- 当前数据库已经满足延迟目标，尚无延迟、吞吐、连接数、扫描量或资源争用指标支持升级。
- 同一服务和同一数据库能提供更简单的同步可见性及故障恢复；查询失败按普通数据库查询重试或返回失败，不需要投影恢复链路。
- 第 2 级已经隔离了分页 DTO/SQL 与聚合写模型，获得本场景需要的职责分离；更高等级只会提前引入同步维护、最终一致、复制和额外运维成本。

## 一致性与事务影响

当前查询直接读取主数据库：写事务提交后，新状态按数据库隔离级别对后续查询可见，不引入额外的投影陈旧窗口。不要承诺未提交写入可被分页查询读取。查询用例保持只读；业务写事务仍由 Application 写用例控制，Query Adapter 不扩大写事务。

如果未来升级到第 4 级，必须先由业务给出可接受的明确陈旧窗口和写后读语义；未给出之前，不得用“最终一致”代替决策。

## 当前实现与验证

- `OrderQueryPort` 由分页查询用例这一真正消费者拥有，Infra 查询适配器实现它。
- 返回专用分页读 DTO，不向 Domain 暴露 DO、Mapper 或页面字段，也不让读 DTO 反向进入写路径。
- 先用查询适配器集成测试验证筛选、稳定排序、页边界、空页和并发插入时的分页语义。
- 用实际 SQL 执行计划和压测记录 P95/P99 延迟、吞吐、扫描行数、连接数及 CPU/IO。只有这些指标越过约定目标，且索引、SQL 或分页策略优化仍不足时，才重新评审第 3 级；不得因“有分页”直接跳到第 4 或第 5 级。
