**分类结论：属于 Event Sourcing 档位。** 资金账户的事件流是聚合事实来源，余额是由事件异步构建的查询投影，因此应应用 Event Sourcing 的完整正确性清单，而不是只套本地 Domain Event 或可靠 Integration Event 清单。

需要审查：

1. **事件流并发与命令幂等**：每个账户流必须有单调版本；追加事件时携带 `expectedVersion`，用乐观并发阻止两个并发命令基于同一旧状态同时成功。冲突后应从最新事件重新构建账户并重新执行业务规则，而不是盲目重试追加。命令还要有稳定幂等键，防止客户端重试产生重复资金效果。
2. **事件应用与流完整性**：事件应用函数必须确定、无外部副作用；同一事件序列每次都重建出相同账户状态。重放必须能检测版本缺口、重复、损坏和未知事件类型，不能静默跳过。账户流内顺序必须明确并被存储层保证。
3. **投影幂等与进度**：余额投影使用事件 ID 或全局/分区位置去重；投影更新与检查点推进应原子提交，避免“余额已改但位置未记”或“位置已记但余额未改”。明确重复、乱序和失败重试策略，并监控投影延迟、停滞、错误率及检查点。
4. **从头重建**：必须能不依赖现有投影，从事件流零点构建新的影子投影；重建期间继续追平增量，校验账户数量、余额和最终检查点后再切换。保留旧投影用于回退，不能把人工修表当作唯一恢复路径。
5. **快照边界**：快照只是条件化性能优化，不是事实来源。删除所有快照后仍必须能从头重建；快照要绑定流版本并验证兼容性，过期或损坏时回退到事件流。
6. **事件契约演进**：事件类型和 schema 显式版本化；为旧事件提供兼容读取或 upcaster，必要的历史迁移必须可审计、可回滚。使用历史事件样本做契约测试，确保新代码仍能读取并重放全部旧版本；禁止原地改变旧事件含义。
7. **查询一致性语义**：投影允许短暂延迟，但必须定义可接受的最大延迟、如何向调用方暴露陈旧性，以及写后立即查余额时是等待投影追到指定流版本、返回带版本的旧值，还是走其他明确路径。不要把最终一致误报为实时余额。
8. **长期运维**：评估完整流重放时间、事件存储容量、备份恢复、投影并行度和重建对线上写入的影响；监控必须能定位到具体账户流、事件位置和失败版本。

关键验证包括：并发命令只有一个以同一 `expectedVersion` 追加成功；相同命令重试只产生一次资金效果；事件全量重放得到相同账户状态；投影重复消费和故障重启后余额不重复；注入缺口、乱序及未知版本时能显式失败；从空投影完成“全量重建—追平增量—校验—切换”；所有历史 schema 样本均可被当前代码重放。

不因采用 Event Sourcing 就自动要求 Outbox 或把内部事件直接当外部 Integration Event。若投影直接可靠地追踪事件存储，可用事件位置和检查点恢复；若未来要跨进程向其他上下文保证最终送达，再单独按可靠 Integration Event 档位审查外部契约、Outbox/发布进度、消费者幂等与失败恢复。
