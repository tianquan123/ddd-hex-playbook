推荐把“商品目录”上下文组织成 **简单 CRUD / Transaction Script**，并在查询侧采用 **CQRS 第 2 级：分离 Command/Query 接口**。这是该上下文的局部架构选择，不要求在内部复制现有六个 Maven 模块的六层目录。

**业务证据**：后台只编辑名称、描述和展示状态；字段之间没有跨字段不变式，也没有复杂生命周期。因此写路径由 Application 用例直接完成输入校验、持久化和本地事务即可，不强造富聚合、值对象层级、Domain Service 或 Domain Event。查询是主要负载，按列表、详情等查询用例定义 Query 接口，直接返回面向页面的 DTO，不为读取重建领域对象。

**职责与依赖**：入口适配器调用 Application 的编辑用例或查询接口；编辑用例消费持久化端口，查询用例消费独立 Query Port；基础设施分别实现写持久化和查询。Domain 保持框架无关是架构不变量，但当前业务甚至可以只有很薄的数据模型，不需要为了形式填满 Domain。端口由实际消费者拥有。

**事务与一致性**：单次编辑在一个本地事务内更新一个商品；查询同步读取同一数据库，事务提交后即可读到新值，一致性窗口为零（以数据库提交可见性为准）。查询失败按普通同步请求重试，Schema 通过现有数据库迁移流程演进。

不选择富领域模型，因为没有需要聚合保护的不变式或生命周期；不选择 CQRS 第 3 级及以上，因为尚无证据表明查询形状、索引组合或负载需要独立读模型，更没有接受陈旧数据、异步投影、独立读库或 Event Sourcing 的业务与运维依据。若以后出现跨字段规则或复杂状态流，再升级写侧模型；若监控显示查询延迟、扫描量或资源争用超标，且专用查询结构有明确收益，再评估第 3 级。

验证重点：用例测试覆盖名称、描述、展示状态的编辑与事务回滚；查询测试覆盖列表、详情、分页和 DTO 映射；集成测试确认提交后的写后读语义；用 `mvn dependency:tree` 与现有依赖守卫确认实际依赖方向，避免为这个上下文新增无边界收益的模块。
