**分类结论：选择“本地 Domain Event”档位。** 不需要 Outbox 和 MQ，也不属于可靠 Integration Event 或 Event Sourcing。

依据是：消费者与订单在同一进程；统计不参与交易正确性；允许偶尔丢失且可离线修复；只要求订单事务提交后再处理。这正是同进程、弱可靠的派生工作的适用场景。Outbox + MQ 会额外引入 relay、重试、幂等、积压监控等可靠投递成本，但当前没有“必须最终送达”或跨进程证据；事件流也不是订单的事实来源，因此不应用 Event Sourcing 的版本、快照、回放和投影重建清单。

只按本地 Domain Event 档位检查：

1. **事实与所有者**：用通用语言命名为类似 `OrderConfirmed` 的已发生事实，由 `Order` 聚合在确认成功时记录；不要命名成 `UpdateStatisticsMessage` 这类技术命令。
2. **来源行为**：只有订单确认领域行为成功后才产生该事件；确认失败或事务回滚不能触发统计更新。
3. **技术无关**：Domain Event 不依赖 Spring、MQ、数据库 DO 等类型，只携带统计确实需要的稳定业务信息，例如订单标识和确认时间。
4. **事务时序与失败语义**：Application 在订单事务仍活跃时保存聚合并发布 Spring 事件，由 `@TransactionalEventListener(AFTER_COMMIT)` 在提交后处理。不要等到 `afterCommit()` 中才首次发布，再期待默认的事务事件监听器接收。统计写入应使用自己的事务；监听失败只记录可观测信息并留给离线修复，不能回滚已经提交的订单，也不能被解释为订单确认失败。

建议流程：

```text
订单事务中：确认订单 -> 保存订单 -> 发布本地事件 -> 提交
提交后：AFTER_COMMIT 监听器 -> 在独立事务中更新非关键统计
```

验证应覆盖：发布发生在活跃事务内；订单事务回滚时监听器不执行；提交后才更新统计；监听器失败时订单仍保持已确认；离线修复能从订单事实数据重算统计。只有未来变成跨进程或统计不可丢失时，才升级评估 Integration Event + Outbox；当前不执行该档位及 Event Sourcing 的检查项。
