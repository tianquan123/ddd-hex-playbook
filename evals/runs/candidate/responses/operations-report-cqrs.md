# 运营报表 CQRS 决策包

## 决策结论

当前选择 **CQRS 第 4 级：异步投影读模型**，但读写侧暂时**共用数据库基础设施**，不进入第 5 级独立读写存储。

这是满足现有证据的最小充分级别：运营报表需要组合订单、库存和履约数据，形状与各写模型明显不同；查询量明显高于写入量；业务明确接受最多五分钟陈旧；报表可以从事实来源重建；团队能运维一个投影任务。采用异步投影可避免订单、库存、履约的写事务同步维护一个跨边界报表结构，同时不引入独立数据库的复制、容量和备份成本。

本结论是**条件化方案**：第 4 级的关键前提是运营方确实接受五分钟窗口内的写后读不一致，并且用于重建的事实来源完整、稳定、可定位进度。若这些前提不成立，应退回第 3 级或重新澄清数据契约，而不是由工程实现补造语义。

## 六级梯度判断

| 级别 | 本场景判断 | 证据与停止/排除理由 |
| --- | --- | --- |
| 1. 同一模型分离命令与查询方法 | 不足 | 报表需要跨订单、库存、履约组合，且结构与写模型完全不同；继续加载写模型会让查询形状受写侧结构约束。 |
| 2. 分离 Command/Query 接口 | 不足 | 独立 Query Port 和 DTO 是必要职责边界，但仅做接口分离仍不能消除跨模型组合、重复扫描与高查询量对写模型的影响。 |
| 3. 写模型与读模型分离、同步同库 | 不选为当前级别 | 它能提供专用读表/视图，但同步维护会把跨订单、库存、履约的报表更新带入写路径或同步协调；业务既允许五分钟延迟，团队也能运维投影任务，没有必要承担这种写侧耦合。若事实来源无法可靠重放，或报表必须事务内立即可见，则应回退到此级重新设计。 |
| 4. 异步投影读模型 | **当前级别** | 不同查询形状、明显读多写少、五分钟陈旧窗口、可重建以及投影运维能力共同支持该级别。读模型可使用同一数据库中的独立表或 schema；“异步”不等于“独立数据库”。 |
| 5. 独立读写存储 | 暂不选择 | 尚无证据表明共享数据库在吞吐、连接数、扫描量、资源争用、独立扩缩容或故障隔离方面达不到目标；团队明确不希望立即拆独立数据库。 |
| 6. Event Sourcing | 排除 | 没有证据要求事件流成为事实来源，也没有审计、时态查询或任意时间回放的业务要求。报表可重建只证明需要可恢复投影，不证明需要 Event Sourcing。 |

## 当前局部架构与职责

- **Application Query Port** 由运营报表查询用例拥有，返回面向报表的只读 DTO，不暴露订单、库存或履约聚合。
- **投影任务** 是入站 Job/消息消费者适配器，读取各来源提供的稳定增量事实，经应用层投影用例写入运营报表读模型。
- **投影写入端口** 由投影应用用例拥有；基础设施适配器负责读表、检查点和影子投影的持久化。
- **查询适配器** 直接按运营报表形状读取专用读模型；不得为了复用而在查询时重建或修改写聚合。
- 订单、库存、履约仍各自拥有写侧事实与事务。运营读模型是派生数据，不成为这些业务事实的新权威。

这是局部 CQRS，不要求整个系统改造成消息驱动架构，也不决定 Maven 模块数量、数据库产品或部署单元。

## 读模型与一致性语义

运营报表读模型应按实际报表维度预组合订单、库存和履约字段，例如以订单及其明细为主键，带上最近已投影的库存与履约摘要；最终列、粒度和聚合维度由报表契约确认，不复制完整可变聚合快照。

- **一致性窗口**：从来源事实可供投影读取起，报表最迟五分钟反映该事实。
- **写后读语义**：写操作成功后立即查询报表，允许看到旧值；接口应返回报表数据时间或投影水位，使使用者能判断新鲜度。
- **超窗语义**：超过五分钟仍未追平不是可接受陈旧，而是投影故障，需要告警；不得静默返回数据并伪装为最新。
- **跨来源水位**：报表需暴露或内部记录订单、库存、履约各自的处理水位，避免单一“更新时间”掩盖某一来源落后。
- **查询事务**：查询只访问读模型，不参与三个写模型的事务，也不获得跨来源瞬时快照保证。

## 投影更新与恢复契约

第 4 级必须把恢复设计成正常能力，而不是依赖人工改表。

1. **事实来源**：明确订单、库存、履约各自可重放的来源，以及事件/变更记录的保留期。来源必须拥有稳定业务标识、版本或单调位置；该来源形式尚未指定，不能默认一定是 MQ。
2. **检查点**：按来源或分区持久化最后成功处理的位置。只有对应读模型变更成功后才能推进检查点；检查点必须可审计。
3. **幂等键**：使用“来源 + 分区 + 事实标识/版本”或等价稳定键去重。重复投递不得重复累计库存、订单数或履约数量。
4. **乱序策略**：按业务对象版本、来源序号或发生时间拒绝旧版本覆盖新版本；跨来源没有全局顺序时，各自推进水位，并通过可重复计算的合并规则形成报表行。
5. **失败重试**：短暂依赖失败可自动重试；不可解析或违反契约的数据进入隔离队列/失败清单，并保留原始标识、原因和重放入口。不得因单条坏数据无限阻塞所有分区。
6. **监控告警**：至少监控各来源检查点、处理延迟、积压量、吞吐、失败率、重试次数、隔离数据量、重建进度和查询延迟。任一来源端到端延迟接近或超过五分钟即告警。
7. **重建流程**：创建版本化的影子读模型；从各事实来源的确定起点全量重放；在重放过程中持续追平增量；校验行数、业务主键覆盖率、关键指标与来源水位；验证通过后原子切换查询指向；保留可回滚窗口，再清理旧投影。
8. **灾难恢复**：投影表丢失或逻辑污染时，从事实来源和检查点重新生成，不把人工修表作为唯一手段。若来源保留期不足以完成重建，必须补充权威快照加增量机制，否则第 4 级恢复前提不成立。

## 事务、依赖与迁移影响

- 写侧业务事务不包含运营报表更新；投影更新与来源写入之间是最终一致性，最大业务容忍窗口为五分钟。
- 投影消费与读模型/检查点更新应具有明确的原子边界：要么该批次的读模型变化和检查点一起成功，要么可安全重放。
- 报表 schema 独立版本化。破坏性变更采用“新影子投影构建—追平—校验—切换”，避免同时要求三个写模型停机迁移。
- 读模型可与写模型暂时使用同一数据库实例，但使用独立表或 schema、独立访问路径和可识别的资源指标，避免形成对写侧表结构的隐式依赖。
- 查询压力必须通过指标观察；共享数据库不代表无限共享资源，应为慢查询、连接池、扫描量和索引膨胀设置容量阈值。

## 运维成本

当前新增并接受的成本：

- 一个可部署、可暂停、可重放的投影任务。
- 三类来源契约及其兼容性管理。
- 检查点、幂等、乱序、失败隔离与重试机制。
- 投影延迟、积压、错误和数据新鲜度的监控告警。
- 影子投影重建、校验、切换和回滚演练。
- 同库报表表、索引及查询资源的容量治理。

当前刻意不承担的第 5 级成本：独立数据库复制链路、双存储容量规划、独立备份恢复、网络与权限治理、跨存储一致性排障和独立值班面。也不承担第 6 级的事件版本长期治理、事件存储迁移及以事件流为唯一事实源的成本。

## 升级与回退触发条件

只有出现可观察证据时才从第 4 级升级到第 5 级，例如：

- 报表查询导致写库连接、CPU、IO、锁等待或扫描量持续越过已约定阈值。
- 报表需要独立扩缩容、发布、备份周期、存储引擎或故障隔离。
- 为满足报表 SLA 必须让读侧在写侧存储故障时继续服务。

在升级前应先验证索引、读模型粒度、查询计划、连接池隔离和容量是否已经合理；不能只凭“查询量高”直接拆库。

若监控显示投影无法稳定在五分钟内追平、事实来源不能覆盖重建、或团队无法承担恢复演练，应停止第 4 级落地：先修复来源与运维能力；若业务又要求立即一致，则回退到第 3 级并重新评估同步维护成本。

## 停止理由

本次明确停在第 4 级。它已经满足不同读模型形状、高读负载、五分钟一致性窗口和可重建要求，也与团队“能运维一个投影任务、暂不拆独立数据库”的能力边界匹配。第 5 级缺少资源争用、独立扩缩容或故障隔离证据，第 6 级缺少事件流成为事实来源的业务理由；继续升级只会增加尚无收益证据的运维复杂度。

## 验证清单

- 自动化测试：同一事实重复两次时指标不重复累计；旧版本晚到时不覆盖新版本；单条坏数据被隔离且不阻塞其他分区。
- 恢复演练：删除一套非生产投影，从来源重建到影子表，追平增量，完成校验和切换，全程无需人工改业务数据。
- SLA 测试：在预期峰值写入和报表查询量下，分别记录投影端到端延迟、积压、吞吐、查询 P95/P99、数据库 CPU/IO/连接数和写侧延迟，确认投影在五分钟内追平且不破坏写侧目标。
- 故障测试：暂停投影超过五分钟，确认告警触发、报表暴露数据水位，恢复后可从检查点安全追平。
- 迁移测试：新增/变更报表字段时能够并行构建版本化影子投影，校验后切换并可回滚。
- 契约测试：订单、库存、履约来源的标识、版本、删除/更正语义和兼容策略均可被投影独立验证。

示例验证命令（具体构建工具和测试名应按项目现状替换）：

```bash
./mvnw test -Dtest='*Projection*Test,*OperationsReport*Test'
./mvnw verify -Pintegration-test
```
