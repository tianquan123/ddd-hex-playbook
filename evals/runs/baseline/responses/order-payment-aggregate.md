# 结构化建模决策包

## 1. 建议结论

**Order 和 Payment 不应放在同一个聚合。**

推荐把它们建模为两个独立聚合：

- `Order` 聚合负责订单确认、取消资格和订单生命周期。
- `Payment` 聚合负责支付尝试、成功、失败及支付侧补偿；退款若有独立规则和生命周期，还可进一步成为独立的 `Refund` 聚合。
- 聚合之间只保存稳定身份引用，例如 `orderId`、`paymentId`，不持有可直接修改的对象引用。
- 跨聚合流程通过已经发生的领域事实和状态推进协调，默认接受最终一致；具体采用 Saga、流程管理器或其他机制，留给工程设计决定。

“订单确认后可以等待支付”已经表明，确认订单并不要求支付在同一原子操作中成功。支付可能晚到、失败或独立补偿，又说明 Payment 有独立生命周期和失败语义。订单取消与退款受不同规则控制，则进一步排除了“取消订单必然等于退款”这一单聚合内联动规则。

## 2. 场景与业务时间边界

### 场景 A：确认订单，尚未支付

- 命令：`ConfirmOrder`
- 成功结果：订单进入已确认、待支付状态。
- 允许状态：在一段业务允许的时间内，订单已确认而 Payment 尚不存在或仍在处理中。
- 结论：订单确认与支付成功之间不存在必须在命令返回前成立的即时一致性要求。

### 场景 B：支付晚到

- 支付可能在订单确认后的任意允许时刻成功。
- Payment 应根据自身幂等规则记录支付结果。
- Order 是否仍接受该支付带来的状态推进，取决于支付到达时订单是否仍处于可支付状态。
- 若订单已经取消，不能通过扩大聚合来隐藏冲突，必须执行明确的晚到支付规则，例如自动退款、人工例外处理或拒绝入账。

### 场景 C：支付失败

- 支付失败不应回滚已经成功确认的订单。
- Order 可以继续等待下一次支付、在超时后关闭，或由业务规则取消。
- Payment 保留失败原因、支付尝试及是否允许重试等自身事实。

### 场景 D：订单取消与退款

- `CancelOrder` 判断订单是否允许取消。
- `RequestRefund` 或等价行为判断一笔已支付资金是否允许退款、退款多少以及退向何处。
- 两者可能相关，但不是同一业务动作：订单可以在未支付时取消而无需退款；支付也可能因独立原因补偿，而订单未必取消。

## 3. 不变式与一致性分类

### Order 聚合内必须即时成立

- 只有满足订单取消规则的订单才能被取消。
- 已取消订单不能再次确认或进入正常履约流程。
- 订单状态迁移必须符合订单生命周期规则。
- 同一订单命令的重复执行不能造成重复状态迁移。

### Payment 聚合内必须即时成立

- 同一次支付意图或支付业务键不能被重复记为多次成功。
- 支付成功、失败、补偿等状态迁移必须符合支付生命周期规则。
- 已完成的补偿不能被重复执行产生第二次资金效果。
- 支付金额、币种及其关联的 `orderId` 在需要保持稳定后不得被任意改变。

### 跨聚合允许延迟或补偿

- 订单已确认而支付仍未完成。
- 支付成功事实到达 Order 之前，Order 的支付视图暂时滞后。
- 订单取消与退款之间允许存在处理中状态。
- 晚到支付、重复通知或流程中断可通过幂等、重试和补偿恢复。

目前没有业务事实证明“Order 的状态与 Payment 的状态必须在一次原子提交中同步变化”。若未来提出这种要求，必须明确是哪条具名业务规则要求即时一致，以及为什么等待、拒绝或补偿都不可接受，再重新评估边界。

## 4. 候选模型

### 聚合候选

```text
Order
  identity: orderId
  owns: 订单项、订单金额快照、订单生命周期
  references: paymentId（可选）或仅关联支付事实的标识

Payment
  identity: paymentId
  owns: 支付意图、尝试、资金结果、支付生命周期
  references: orderId

Refund（条件化候选）
  identity: refundId
  owns: 退款申请、金额、退款状态、失败与重试业务语义
  references: paymentId、orderId
```

是否把 Order 与 Payment 放在同一个限界上下文，现有事实还不足以确定。若订单规则与资金规则由不同业务权威制定，语言、合规约束、发布节奏或事实来源也不同，应进一步考虑 `Ordering` 与 `Payments` 两个限界上下文。即使暂时位于同一应用或数据库，它们仍应保持独立聚合。

### 领域事件候选

- `OrderConfirmed`
- `OrderCancelled`
- `PaymentSucceeded`
- `PaymentFailed`
- `RefundRequested`
- `RefundCompleted`
- `RefundFailed`

事件表达已经发生的业务事实，不表达“请执行退款”之类的未来意图。生产事件的聚合不应依赖具体消费者，也不应在事件中携带完整、可变的聚合快照。

## 5. 跨聚合协作规则

建议把流程明确建模为状态推进，而不是伪装成一个大事务：

1. Order 成功确认，产生 `OrderConfirmed`。
2. 支付流程接受支付请求，Payment 独立成功或失败。
3. `PaymentSucceeded` 到达后，Order 仅在当前状态允许时推进为已支付；重复事实不得产生第二次业务效果。
4. 若 `PaymentSucceeded` 晚于 `OrderCancelled`：
   - 若业务规定必须自动退回，则启动退款流程；
   - 若不能自动退款，则进入明确的异常处理状态；
   - 不得悄悄把已取消订单恢复为正常订单。
5. 订单取消若涉及已成功支付，只表达订单侧取消事实，再由独立退款规则决定是否、何时以及退多少。

这里的关键不是选择某个流程实现模式，而是明确：每个聚合只守护自己的即时不变式，跨聚合冲突必须有可命名、可观察、可恢复的业务结果。

## 6. 反例与边界挑战

- **重复支付成功通知**：Payment 和 Order 都不得重复产生业务效果。
- **支付成功与订单取消并发**：结果不能依赖数据库提交先后偶然决定，必须定义晚到支付策略。
- **订单取消后支付失败通知到达**：订单保持取消，失败事实可记录但不应反向改变订单。
- **退款成功但订单取消失败**：证明退款与取消不能假定为同一原子动作，需要补偿或人工处理状态。
- **订单删除或归档**：Payment 的历史与补偿能力不能因 Order 生命周期结束而丢失。
- **多次支付尝试**：需要明确一个订单是否允许多个 Payment、多个尝试属于一个 Payment 还是多个 Payment；无论选择哪种，都不能靠把 Payment 塞进 Order 来规避身份与幂等规则。

## 7. 被排除的候选方案

### 候选：Order 作为根，Payment 作为其内部实体

排除原因：

- Payment 的成功、失败、晚到和独立补偿构成独立生命周期。
- 订单确认不等待支付，二者没有共同的即时一致性边界。
- 订单取消和退款规则不同，无法由一个简单的根状态迁移统一守护。
- 大聚合会把支付重试、外部资金结果和订单并发修改无谓地耦合在同一冲突范围内。

### 候选：Payment 作为根并拥有 Order

排除原因：订单可以先独立确认和等待支付，Order 并非 Payment 生命周期的组成部分。

### 候选：用 DomainService 在一个事务中同时保存两个聚合

排除原因：DomainService 不能掩盖未证明的跨聚合即时一致性要求。现有场景明确允许等待、失败和补偿，默认应采用独立事务与流程协调。

## 8. 假设与尚未解决的问题

当前结论基于以下假设：

- 订单确认不代表资金已经成功结算。
- 短暂的订单状态与支付状态不一致对业务是可接受的。
- 支付失败或退款失败可以重试、补偿或进入人工处理。
- 没有法规或资金账务规则要求 Order 与 Payment 共享单次原子提交。

仍需业务确认的问题：

1. 一个订单允许多少次支付意图和支付尝试？
2. 订单取消与支付成功并发时，以哪个业务时点判断，晚到支付必须自动退款还是进入人工处理？
3. 部分支付、部分退款和多币种是否存在？
4. “已支付”由 Payment 的哪个事实定义：渠道授权、资金捕获还是最终结算？
5. Order 是否需要保存支付状态的派生视图，还是只在查询模型中组合？
6. 订单与资金规则是否由不同业务权威负责，从而需要拆分限界上下文？

这些答案可能细化 Payment、Refund 和上下文边界，但在没有新增“跨对象即时不变式”的情况下，不会改变 Order 与 Payment 分属不同聚合的结论。

## 9. 供 Java/Spring 架构评审使用的工程交接契约

以下是建模侧交给工程评审的约束，不规定具体 Java 包名、Spring 注解或消息技术：

### 模型边界

- 必须保留 `Order`、`Payment` 两个聚合根；`Refund` 是否独立建根由退款生命周期问题确认后决定。
- 一个应用事务默认只修改一个聚合。
- 聚合之间仅通过身份和值传递协作，不共享可变实体对象。
- Repository 的加载和保存边界必须与聚合边界一致，不能通过级联保存重新制造大聚合。

### 命令与事实

- 订单侧命令至少区分确认与取消；支付侧命令至少区分支付处理与退款请求。
- `OrderConfirmed`、`OrderCancelled`、`PaymentSucceeded`、`PaymentFailed` 等是过去时业务事实。
- 领域模型不得依赖 Spring publisher、MQ client 或其他技术发布器。
- 跨边界契约若需要发布，应由工程方案把内部领域事实转换成稳定的外部契约，而不是直接暴露聚合对象。

### 一致性与流程

- 不得用跨两个 Repository 的默认本地事务假装 Order 与 Payment 原子一致。
- 应用层需要显式承载订单—支付—退款的长流程、状态推进和失败恢复。
- 必须为支付晚到、重复、失败、取消并发和退款失败定义可观察的处理结果。
- 若工程评审主张跨聚合原子更新，必须回到业务侧给出具名即时不变式，并说明为何延迟或补偿不可接受。

### 幂等与并发验收

- 重复的支付成功事实只允许一次资金效果和一次订单状态推进。
- `PaymentSucceeded` 与 `CancelOrder` 并发时，结果必须符合已确认的晚到支付规则。
- 退款重复触发不能产生重复退款。
- 单个聚合的并发写必须检测冲突；不能用扩大聚合范围解决跨聚合竞争。

### 最低测试契约

- 订单确认后无 Payment，Order 仍处于合法状态。
- 支付失败不会回滚或破坏已确认订单。
- 支付成功晚于订单取消时，系统进入规定的退款或异常路径。
- 订单取消不自动等同于退款成功。
- 重复、延迟及乱序的支付事实不会产生重复业务效果。
- 任一流程步骤失败后，可以从持久化的业务状态继续，而不是依赖一次进程内调用链完成。

架构评审可以自由选择事务、消息、重试和持久化实现，但实现不得破坏上述聚合边界、事实语义与一致性分类。
