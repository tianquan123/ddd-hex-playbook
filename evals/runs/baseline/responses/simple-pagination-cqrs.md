## 结论

当前应停在 **CQRS 决策梯度第 2 级：代码职责上的读写分离，同一服务、同一数据库**。不需要异步投影，也不需要独立读库。

实现上，由 Application 拥有 `OrderQueryPort`，分页查询直接返回 `OrderSummary` 等用例读模型；Infra 的查询适配器针对订单表执行分页和少量筛选。不要为了列表查询加载、重建订单聚合，也不要让分页 DTO、SQL 条件或数据库类型进入 Domain。

## 六级 CQRS 决策梯度

| 级别 | 策略 | 适用信号 / 升级触发条件 |
|---|---|---|
| 1 | 单一模型直接查询：读写都通过聚合 Repository | 查询很少且字段与聚合加载天然一致；一旦分页、列表投影开始污染 Repository，就升级 |
| **2（当前）** | **逻辑读写分离：独立 Query Port 和读模型，但同服务、同库、同步读取原表** | **列表/分页需要专用字段和筛选，但查询简单，数据库已满足延迟目标** |
| 3 | 同库专用读结构：视图、覆盖索引或同步维护的读表 | 多表连接、计算或扫描持续导致慢查询，常规索引和 SQL 优化仍无法达标；同时业务仍要求读后立即可见 |
| 4 | 同库异步投影：写模型提交后异步更新读模型 | 同步维护明显拖慢写事务，且业务明确接受可量化的短暂陈旧；团队能承担重放、追平和幂等处理 |
| 5 | 独立读库 + 异步投影 | 读流量、连接数、存储布局或故障隔离已影响写库，必须独立扩缩容；业务接受最终一致性 |
| 6 | 独立读服务 / 多种投影 | 查询需要独立发布、独立 SLA、多个存储引擎或多个面向不同消费者的投影，组织和运维收益超过分布式成本 |

这六级是决策梯度，不是必须逐级建设的成熟度模型。只有出现可观测的瓶颈或明确的一致性需求变化时才升级。

## 为什么停在第 2 级

- 查询只涉及订单表和少量筛选，没有跨聚合拼装或昂贵计算。
- 当前数据库已经满足延迟目标，没有性能证据支持复制数据或增加投影链路。
- 同服务、同数据库可保持简单的读后写一致性，不需要定义“允许陈旧多久”、投影延迟或追平策略。
- Query Port 已经把分页读模型与聚合 Repository 分开，获得当前真正需要的 CQRS 收益，同时不引入消息、Outbox、重复消费、重放和独立读库运维。

## 规则级别、所有者与影响

| 决定 | 规则级别 | 所有者 / 影响 |
|---|---|---|
| 分页查询由 Application Query Port 表达 | 架构不变量 | Application 是端口消费者和所有者 |
| Query Adapter 直接查询订单表并返回用例读模型 | 团队默认值 | Infra 实现；Domain 不感知 DO、SQL 和分页展示字段 |
| 聚合 Repository 与 Query Port 分离 | 架构不变量 | 写路径继续保护聚合；读路径不重建聚合 |
| 读写继续共用服务和数据库 | 当前条件化方案 | 最小部署与运维成本；读取可见性与数据库事务语义一致 |
| 暂不使用异步投影和独立读库 | 当前推荐 | 不产生最终一致性、投影修复和消息可靠性成本 |

## 升级时必须补齐的设计

- 升到第 3 级前：先用执行计划、慢查询、索引命中率和真实分页深度证明瓶颈；优先尝试合适索引、稳定排序和游标分页。
- 升到第 4 级前：明确允许陈旧时间、投影幂等键、乱序处理、重放/重建、积压告警和失败恢复。若通过消息可靠更新投影，业务写入与 Outbox 应同事务提交。
- 升到第 5 级前：证明读负载确实需要独立扩缩容或故障隔离，并定义读库不可用、落后和切换时的产品行为。
- 升到第 6 级前：证明独立团队、发布节奏或多种查询存储确有长期价值，否则只是把一次 SQL 查询变成分布式系统。

## 未选择的方案及原因

- **异步投影**：当前没有同步读路径的性能瓶颈，也没有接受陈旧数据的业务必要性。
- **独立读库**：当前读写负载未形成资源争用或隔离需求，复制延迟与运维成本没有收益支撑。
- **所有查询通过订单 Repository**：会把分页筛选和展示读模型带入聚合持久化契约，职责不清。
- **把“使用 Query Port”直接等同于完整 CQRS**：逻辑读写分离可以是局部代码策略，不要求消息、双库或独立部署。

## 验证方式

- Query Adapter 集成测试：覆盖筛选组合、稳定排序、页边界、空结果和大偏移量或游标行为。
- SQL 验证：对代表性数据量执行 `EXPLAIN`/`EXPLAIN ANALYZE`，记录 P95/P99 延迟、扫描行数和索引使用情况。
- 架构测试：确保 Domain/Application 不依赖 DO、DAO 或数据库分页类型，查询读模型不进入聚合写路径。
- 容量门槛：以实际 SLO、数据库 CPU/IO、连接池饱和度和慢查询比例作为升级证据，而不是以订单量或“用了 CQRS”作为依据。
