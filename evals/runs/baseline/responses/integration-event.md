先分类：

| 档位 | 是否适用 | 判断 |
|---|---:|---|
| 本地 Domain Event | 仅作为订单上下文内部事实适用 | `Order` 聚合可以记录 `OrderConfirmed` Domain Event，但它本身不具备跨进程可靠投递能力。 |
| 可靠 Integration Event | **本题应选** | 订单与履约独立部署，代理至少一次投递，代理故障后仍须最终发送，因此必须使用 **业务事务 + Outbox + 幂等消费者**。 |
| Event Sourcing | 不选 | Event Sourcing 是以事件流重建聚合的持久化模型，不是可靠发消息的必要条件；除非订单本身还有审计、时态查询或事件回放等明确需求。 |

完整设计如下。

```text
确认订单的应用事务
  -> Order.confirm()
  -> 聚合记录内部 DomainEvent
  -> 保存 Order
  -> 映射为版本化 OrderConfirmed IntegrationEvent
  -> 插入 Outbox
  -> 同一数据库事务提交

Outbox Relay
  -> 竞争领取未发布记录
  -> 发布到 broker
  -> 成功后标记已发布
  -> 失败则按策略重试

履约 MQ Trigger
  -> 反序列化和协议校验
  -> 调用“创建履约任务”应用用例

履约应用事务
  -> 检查并写入幂等记录
  -> 创建履约任务
  -> 提交后 ack
```

关键检查项：

1. **事务原子性**

   订单状态和 Outbox 必须在同一个本地数据库事务中提交。订单事务回滚时不能留下可投递消息；订单提交成功而代理不可用时，Outbox 记录仍然存在，relay 恢复后继续发送。

2. **事件边界**

   Domain Event 属于订单领域内部，保持技术无关；Integration Event 是订单上下文对外发布的稳定契约。不要让同一个 Java 类同时充当内部事件和外部长期契约，也不要直接序列化订单聚合。

   外部事件至少包含：

   - `eventId`
   - `eventType = OrderConfirmed`
   - `schemaVersion`
   - `orderId`
   - `occurredAt`
   - 履约真正需要的订单快照字段
   - 可选的 `correlationId`、`causationId`

3. **发布语义**

   relay 与 broker 之间仍可能出现“broker 已接收，但 Outbox 尚未来得及标记成功”，所以重复发布是正常情况。系统承诺的是最终至少一次送达，不是假设性的端到端 exactly-once。

   relay 需要并发领取机制、退避重试、最大尝试次数、错误原因、积压监控和告警。不可恢复消息进入 DLQ，但不能无声丢弃。

4. **履约幂等**

   履约必须在**同一个业务事务**内完成：

   - 插入已处理事件记录，例如以 `(consumer_name, event_id)` 建唯一约束；
   - 创建履约任务；
   - 提交事务。

   若唯一约束冲突，视为重复消息，直接返回成功。除此之外，履约任务还应以业务键（通常是 `orderId`，必要时加任务类型）建立唯一约束，作为第二道防线。这样才能保证“至少一次投递、一次业务效果”。

   传输去重可以位于 MQ Trigger 或消息基础设施；“一个订单只能创建一次履约任务”的业务幂等必须由 Application 用例负责，其存储通过端口抽象、由 Infra 实现。

5. **ack 与失败处理**

   只有履约事务提交成功后才 ack。反序列化失败、契约不兼容等不可恢复错误应隔离到 DLQ；数据库暂时不可用等可恢复错误应 nack/retry。不能先 ack 再创建任务。

6. **顺序与契约演进**

   如同一订单还有取消、变更等后续事件，应按 `orderId` 分区，只承诺同一订单所需的局部顺序，不承诺全局顺序。事件契约需要版本化，并定义向后/向前兼容策略；新增字段优先保持可选，删除或改义需升级版本。

规则级别：

- **架构不变量**：Domain Event 不等于技术消息；跨上下文可靠投递使用 Integration Event；Domain 不依赖 Spring 或 MQ。
- **本题约束推导出的必需方案**：订单数据与 Outbox 同事务、消费者业务幂等、事务提交后 ack。
- **团队默认值**：按 `orderId` 分区、指数退避、DLQ、积压与失败告警。
- **条件化方案**：只有订单确实需要事件流作为事实来源时才引入 Event Sourcing。

不选择的方案：

- `AFTER_COMMIT` 后直接发 broker：提交后进程崩溃或代理不可用会永久丢消息。
- 数据库与 broker 双写：不存在覆盖两者的本地原子事务。
- 只依赖 broker 去重：至少一次投递仍会重复，且无法保护履约业务效果。
- 为可靠通知引入 Event Sourcing：扩大持久化、重建、版本演进和运维复杂度，却不能替代 Outbox 与消费幂等。

验收测试至少包括：

- 订单事务回滚后，订单未确认且无 Outbox。
- 订单提交成功、broker 停机后，Outbox 保留；broker 恢复后最终发送。
- relay 在“发布成功但标记失败”处崩溃，重启后重复发布，履约仍只创建一个任务。
- 同一事件并发投递两次，只产生一个幂等记录和一个履约任务。
- 履约事务失败时不 ack，重投后可以成功处理。
- DLQ 回放、延迟和同一订单的必要乱序场景不会产生重复业务效果。
- 契约测试验证 `OrderConfirmed` schema 兼容性及敏感字段不会泄露。
