## 推荐方案

在“商品目录”上下文内部采用**轻量应用服务 + 读写路径分离**，而不是为了六个 Maven 模块构造一套富领域模型。

- 查询路径：Application 拥有 `CatalogQueryPort`，直接返回商品详情、列表或分页所需的用例读模型；查询适配器按页面需要读取，不为查询重建商品聚合。
- 编辑路径：用一个很薄的 Application Service 编排“加载或定位商品 → 校验输入 → 更新名称、描述或展示状态 → 持久化”，并控制单次本地事务。
- Domain：只保留确有业务含义的类型和规则，例如 `ProductId`、`DisplayStatus` 及字段自身约束。当前没有跨字段不变式和复杂生命周期，不必引入 DomainService、复杂聚合行为或领域事件。
- 现有六模块只是承载这些职责的物理结构；按项目已有依赖方向放置即可，不要让每个用例机械穿过六层，也不需要为目录上下文新增六个对称模块。

这是一种局部的“简单写模型 + 专用读模型”策略，不等于引入事件溯源、消息总线或完整 CQRS 基础设施。

## 规则级别与所有者

| 决定 | 规则级别 | 所有者 |
|---|---|---|
| Application 编排编辑用例并控制事务 | 架构不变量 | Application Service |
| 查询端口直接返回用例读模型 | 团队默认值 | Application / `CatalogQueryPort` |
| Query Adapter 执行列表、详情、分页查询 | 架构不变量 | Infra，实现 Query Port |
| 名称、描述和展示状态的输入解析与协议映射 | 架构不变量 | Trigger |
| 字段自身的业务约束放在领域类型中 | 条件化方案；仅在确有业务规则时使用 | Domain |
| 不建立 DomainService、事件和复杂聚合生命周期 | 当前场景推荐 | 当前无相应业务证据 |
| 暂不加缓存 | 团队默认值 | 先由数据库满足查询目标，指标不足再引入 |

## 依赖与一致性影响

- Trigger 只依赖 Application 契约；Application 不接收 HTTP Request、持久化 DO 或 `ApiResponse`。
- Application 拥有查询端口和编辑所需的出站端口，Infra 实现它们；Domain 不依赖 Spring、DAO 或查询读模型。
- 一次后台编辑只修改一个商品并在一个本地事务中提交。当前没有跨商品或跨字段的即时一致性要求。
- 读模型不进入写路径。写后查询需要多强的新鲜度，应由产品要求决定；若直接查询同一数据库，通常可获得足够简单的一致性模型。
- 若以后加入缓存，优先放在查询适配器侧，并明确提交后失效、TTL 和允许陈旧时间，不能把缓存塞进聚合 Repository 当作默认行为。

## 未选择的方案

- **富领域模型 / 大量聚合行为**：没有跨字段不变式或复杂生命周期支撑，会把简单编辑包装成无收益的仪式。
- **DomainService**：当前行为可自然归于字段校验或 Application 用例编排，没有独立、稳定且无对象归属的领域能力。
- **所有查询都经 Repository 重建聚合**：商品目录以查询为主，会增加映射与加载成本，也会让分页、筛选和展示字段污染写模型。
- **完整 CQRS、事件驱动或 Event Sourcing**：当前没有独立扩展、审计重放或异步一致性的需求证据。
- **按六个 Maven 模块逐层复制目录**：模块数量不是上下文内部架构策略，机械镜像会制造空壳类型和无意义转换。

## 评审时需要确认的热点

1. “展示状态”是否只有合法枚举校验，还是存在定时展示、审批或渠道差异；后者会改变写模型复杂度。
2. 名称和描述是否有长度、唯一性、语言版本或敏感词规则；唯一性会引入并发冲突处理。
3. 商品列表需要哪些筛选、排序和分页语义，以及是否允许短时间陈旧；这些决定 Query Port 契约和是否需要缓存。
4. 后台并发编辑是否需要乐观锁；若需要，冲突必须返回可识别的用例失败，重试时重新加载数据。

## 可执行验证

- Application 单元测试：用 fake/mock 端口验证编辑字段、事务失败传播及“不调用无关依赖”。
- Query Adapter 集成测试：验证详情、筛选、排序、分页边界和展示状态条件。
- 持久化契约测试：验证商品保存与重新读取，以及启用乐观锁后的并发冲突。
- ArchUnit：禁止 Domain/Application 引用 Spring Web、DO、DAO 或数据库驱动类型。
- 在项目中执行现有的 Maven 测试入口，例如 `mvn test`；若是多模块构建，则从项目根目录运行，避免把模块划分本身当成验证目标。
