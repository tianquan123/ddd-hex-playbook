结论：该场景主档应归为 **Event Sourcing**。账户事件流是事实来源，余额只是允许短暂滞后的可重建投影。Domain Event 负责表达领域事实；若需跨上下文发布，再从已提交事件转换为 Integration Event，不能用后两者替代 Event Sourcing 的正确性保障。

### 三档分类

| 档位 | 本场景定位 | 能否满足要求 |
|---|---|---|
| 本地 Domain Event | 如 `FundsDeposited`、`FundsWithdrawn`，由账户聚合记录，技术无关 | 单独不能。普通本地事件不保证持久化、完整顺序或可从头回放 |
| 可靠 Integration Event | 面向其他上下文的版本化契约，通过 Outbox/可靠订阅发布 | 仅解决跨边界可靠投递，不天然是账户事实来源；不应直接复用内部事件类 |
| Event Sourcing | 每个账户的有序事件流是权威状态，聚合和余额投影均由事件重建 | 符合全部已知事实，应采用本档检查项 |

### 核心正确性检查

**1. 写入与并发——架构不变量**

- 每条账户流必须有单调递增的流版本；追加时携带 `expectedVersion`。
- `(accountId, streamVersion)` 必须唯一，防止两个并发命令都成功覆盖同一前置状态。
- 版本冲突后必须重新加载事件流并重新执行余额、账户状态等业务规则，不能盲目重试原 SQL 或直接再次追加旧结果。
- 命令应有业务幂等键。客户端超时后无法判断追加是否成功时，重复命令不能产生二次扣款。
- 同一命令产生的多个事件应原子追加；失败时不能留下半条业务状态。
- 账户聚合的即时不变式必须在写侧守护。典型例子包括币种一致、冻结账户不可支出、可用余额不得低于允许下限。异步投影不能承担这些校验。

**2. 事件流重建——Event Sourcing 必需项**

- 严格按账户流版本回放，检测缺口、重复和非法顺序。
- 事件应用函数必须确定性执行，不能在回放时读取当前时间、生成随机数、调用远程服务或再次发布副作用；影响结果的数据应记录在事件中。
- 重建应跳过 ID 生成、创建通知等创建期副作用，但不能绕过账户生命周期不变式。
- 未识别的事件类型或版本必须显式失败、隔离或走版本化重建，不能静默忽略。
- 即使未来使用快照，也必须保留“不依赖快照可从头重建”的验证。快照只是性能优化，不是新的事实来源。

**3. 异步余额投影——允许延迟不等于允许错误**

- 投影处理应按至少一次交付设计；重复事件必须无重复余额效果。
- 推荐在同一数据库事务中同时更新余额和投影 checkpoint，或保存 `(projectionName, eventId)` 去重记录。
- 只要求账户内顺序，不应假设全局顺序；若经过消息系统，应以 `accountId` 作为分区/顺序键。
- 必须定义乱序、缺口、毒事件、重试耗尽和 DLQ 回放策略。不能跳过失败事件后继续生成貌似正常的余额。
- 查询契约应明确余额可能陈旧；建议返回投影版本或更新时间。若某用例要求“写后立即读”，需等待投影追平、读取写模型，或明确提供强一致查询，不能假装异步投影已经同步。
- 应定期把投影余额与事件流完整折叠结果做对账。

**4. 从头重建投影**

- 事件保留策略必须保证完整历史可读取；删除旧事件会直接破坏重建能力。
- 新投影应写入独立的版本化表或命名空间，从起始位置回放，再追平实时事件，最后原子切换查询流量。
- checkpoint 必须可清空、可重建、可观察；重建失败后应能够安全续跑或重新开始。
- 重建过程不能污染线上旧投影，也不能因为实时事件与历史回放竞态而丢失事件。
- 同一事件集重复重建应得到相同余额结果。

**5. 事件契约演进**

- 已持久化事件应视为不可变历史；不要直接修改旧事件含义。
- 每种事件应有明确的 schema/version，并测试旧版本事件仍可被当前代码读取。
- 可采用向上转换器、兼容读取器或显式迁移；字段重命名、币种精度变化和语义变化尤其需要版本策略。
- 内部 Event Sourcing 事件与外部 Integration Event 分离。外部契约可从已提交的内部事件转换，不能让外部消费者冻结聚合内部模型。
- 契约测试需要覆盖前向/后向兼容以及敏感字段泄漏。

### 可靠跨边界发布

若余额投影与写模型不共享同一可靠事件订阅机制，使用：

```text
账户命令事务
  -> 校验 expectedVersion
  -> 原子追加账户事件
  -> 写入 Outbox
  -> 提交

Relay
  -> 发布 IntegrationEvent
  -> 失败重试

投影消费者
  -> 去重
  -> 更新余额与 checkpoint
  -> 提交后 ack
```

Relay 到 broker 通常仍是 at-least-once，因此不能承诺 exactly-once，也不能省略投影幂等。若投影直接订阅事件存储的持久化日志，可不额外复制 Outbox，但仍需 checkpoint、重复处理和失败恢复策略。

### 职责归属

- **Domain**：账户不变式、事件语义和事件应用逻辑；不依赖 Spring、MQ 或数据库。
- **Application**：命令编排、事务、业务幂等以及事件存储/查询端口。
- **Infra**：事件存储、乐观并发、Outbox、投影表和 checkpoint 实现。
- **MQ Trigger**：反序列化、ack/nack、分区信息和传输去重；业务幂等仍归 Application。
- **Query Port**：提供余额读模型，不让投影数据反向进入账户写路径。

### 必须执行的验证

- 两个命令基于同一账户版本并发扣款，最多一个按旧版本成功；另一个重新加载后重新校验。
- 命令超时后以相同幂等键重试，不产生二次资金效果。
- 对全部历史事件从零重建账户及余额，结果与线上状态一致。
- 每个事件重复、延迟、乱序及重放后，余额不重复变化。
- 投影事务在更新余额与 checkpoint 之间失败，恢复后不丢事件。
- 使用历史版本事件运行当前读取器，契约兼容测试通过。
- broker/投影暂停后恢复，能够追平且无缺口。
- 可执行项目现有的 `mvn test` 与 `mvn verify`，并确保并发、回放和失败路径测试没有被跳过。
