分类：**本地 Domain Event / 进程内事件档**。

判断依据：

- 消费者与订单在同一进程。
- 统计不参与订单交易正确性。
- 允许偶发丢失，并能离线重算。
- 只要求订单事务提交后执行，不要求可靠投递。

推荐流程：

```text
订单事务内
  -> 保存订单聚合
  -> 发布技术无关的 DomainEvent / Spring 进程内事件
  -> 提交事务
  -> @TransactionalEventListener(AFTER_COMMIT)
  -> 更新非关键统计
```

结论：**不需要 Outbox，也不需要 MQ**。它们用于跨上下文或必须可靠投递的 Integration Event；这里引入只会增加 relay、重试、去重和运维成本。Event Sourcing 也不适用，因为没有通过完整事件流重建订单状态的需求。

当前档位只需检查：

- 事件必须在订单事务仍活跃时发布。
- 统计处理器使用 `@TransactionalEventListener(phase = AFTER_COMMIT)`。
- 不要到 `afterCommit()` 中才首次发布事件，否则默认的事务事件监听器无法绑定已经结束的事务。
- 统计写入若需要数据库事务，应由处理器开启自己的新事务。
- 明确接受进程崩溃或处理失败造成的漏统计，并提供按订单事实重算/修复的离线任务。

建议验证：

- 订单事务回滚时，统计处理器不执行。
- 订单提交成功时，处理器在提交后执行。
- 处理器异常不影响已经提交的订单。
- 人为漏掉事件后，离线重算能恢复正确统计。

规则级别：这是满足“同进程且允许弱可靠性”条件下的**条件化方案**；一旦改为跨进程、跨上下文或要求不可丢失，再升级为 Integration Event + Outbox + MQ，并补充消费者幂等。
