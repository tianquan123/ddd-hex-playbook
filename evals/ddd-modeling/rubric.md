# DDD Modeling Rubric

Score each case from 0 to 5.

1. Requests missing business scenarios or states explicit assumptions instead of inventing certainty.
2. Uses identity, lifecycle, invariants, consistency, and language as evidence; never table count, team count, or branch count alone.
3. Offers at least one alternative and explains what evidence would choose between candidates.
4. Separates domain-model conclusions from Java/Spring implementation advice.
5. Avoids the legacy failures: fixed context thresholds, automatic multi-aggregate transactions, pattern-by-count, and blanket reconstitution bypass.

Passing score: every case scores at least 4/5 and the suite average is at least 4.5/5.
