# CQRS Decision Ladder

## Core rule

选择能满足当前证据的**最小充分复杂度**。CQRS 是按命令与查询需求逐步分离职责的工具，不是成熟度阶梯；Event Sourcing 不是默认终点。大多数系统停在第 1 或第 2 级是正常结果。

## Decision ladder

| 级别 | 方案 | 需要的证据 | 主要新增成本 |
| --- | --- | --- | --- |
| 1 | 在同一模型中分离命令与查询方法 | 读写简单，模型加载成本可接受 | 方法职责和测试边界 |
| 2 | 分离 Command/Query 接口 | 查询形状开始不同，但同步同库已满足目标 | 独立 Query Port、读 DTO 和 SQL |
| 3 | 写模型与读模型分离，但共用数据库 | 查询组合、索引或模型形状明显不同 | 同库读结构、同步维护与迁移 |
| 4 | 异步投影读模型 | 写事务不能承担同步维护，且业务接受明确陈旧 | 幂等消费、检查点、监控和投影恢复 |
| 5 | 独立读写存储 | 读负载、存储形态或故障隔离必须独立 | 复制链路、独立容量、备份和一致性排障 |
| 6 | Event Sourcing | 事件流必须成为事实来源，并有审计、时态查询或回放价值 | 事件版本、重建、投影、迁移和长期运维 |

第 1 级“分离命令与查询方法”和第 2 级“分离 Command/Query 接口”都可以共用数据库。第 3 级开始显式分离读写模型，但仍不要求独立存储。第 4 级才引入异步投影，第 5 级才引入独立读写存储。

## Evidence required

每次建议必须明确：

- 当前级别及当前查询/读模型形状。
- 升级所需的可观察证据，而不是“最佳实践”。
- 可接受的一致性窗口与写后读语义。
- 当前级别的恢复方式和新增运维成本。
- 为什么不选择更高一级，尤其是紧邻的下一级。

性能证据应包含延迟、吞吐、连接数、扫描量或资源争用；组织证据应包含独立发布、扩缩容或故障隔离需求。单纯存在分页、事件或多个 DTO 不是升级证据。

## Projection recovery

第 4 级及以上必须定义投影检查点、幂等键、乱序/重复策略、失败重试、监控、告警和从事实来源重建的方法。重建应能写入影子投影、追平增量、校验后切换；不能把人工修表当作唯一投影恢复方案。

第 1–3 级不应套用异步投影清单，但仍需说明查询失败如何重试、读写是否在同一事务可见以及 schema 迁移方式。

## Stop rule

当前级别满足查询目标、一致性窗口、恢复目标和团队运维能力时立即停止。只有真实指标或业务语义变化触发升级；每次输出都回答“为什么不选择更高一级”。

## Common mistakes

- 把 CQRS 等同于消息总线、异步投影或微服务。
- 为普通分页查询直接拆独立读库。
- 只描述正常更新，不设计检查点、重放与失败恢复。
- 用 Event Sourcing 解决可靠消息投递，或把它当成最终成熟形态。
- 只写收益，不写一致性窗口、恢复路径和运维成本。
