# Bounded Contexts and Ubiquitous Language

## Core idea

模型只在明确边界内拥有稳定含义。边界不是组织图、数据库或部署单元的自动映射，而是某套语言、规则和事实权威保持一致的范围。

## Evidence

- 同一术语在不同业务流程中是否代表不同概念。
- 规则由谁定义、谁有权改变。
- 哪个上下文是某项事实的权威来源。
- 生命周期、合规约束、发布节奏和服务质量是否不同。
- 团队边界与事务边界是证据，不是固定数量的拆分公式。

## Procedure

1. 收集具体命令、结果和领域事件。
2. 记录在不同参与者口中含义不同的术语。
3. 找到每条规则和事实的业务权威。
4. 提出上下文候选，并描述它们如何协作。
5. 用一个跨边界场景验证语言翻译、失败责任与一致性。

上下文候选通过语言、规则、事实权威和跨边界场景探针后，再使用上下文映射方法分析关系方向、契约、翻译和失败责任。这里不预选关系模式；关系标签必须晚于关系事实。

## Scenario probe

如果“账户”在结算中表示资金账户，在身份系统中表示登录主体，要求双方分别说明账户的创建、冻结和关闭规则。若规则和事实权威不同，即使共享一张表，也可能属于不同上下文。

## Common mistakes

- 仅因团队不同就拆分，或仅因共享事务就合并。
- 规定“若三个维度中两个不同就必须拆”。
- 在含义不同的上下文之间共享一个“标准实体”。
- 用 `controller/service/repository` 等技术层名称命名上下文。
- 把当前部署形态当作不可变化的领域边界。
