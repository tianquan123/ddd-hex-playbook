# Aggregates and Invariants

## Core decision

Aggregate 是为保护具名业务不变式而选择的一致性边界。先从最小候选开始，只有某对象必须通过根在同一次原子变更中保持规则时，才把它纳入边界。

## Procedure

1. 把每条不变式写成业务句子，而不是 SQL 或锁策略。
2. 列出判断该规则所需的信息。
3. 明确临时不一致可以容忍多久，以及失败后能否补偿。
4. 提出能守护即时不变式的最小边界。
5. 对独立生命周期的对象只保存其身份引用。
6. 用并发、失败和删除场景挑战候选边界。

## Consistency decision

| 需求 | 候选处理 |
|---|---|
| 规则在命令返回前必须成立 | 聚合内原子修改 |
| 短暂不一致可接受 | 跨聚合事件与状态推进 |
| 失败可补偿 | Saga/流程补偿，由实现层决定机制 |
| 必须跨聚合即时一致 | 作为例外举证，并重新审视聚合边界 |

默认一个事务修改一个聚合。多聚合原子更新不是禁止项，但必须说明：哪条业务规则要求即时一致、为什么不能调整边界、为什么补偿不可接受。不要把未解决的一致性决策藏进 DomainService。

## Scenario example

“创建订单时扣库存”不足以证明 Order 与 Inventory 是同一聚合。应先问：库存是否有独立生命周期和高并发规则；订单接受后是否允许库存预留处于 `PENDING`；失败能否释放预留。答案决定聚合和流程，而不是两张表是否同库。

## Common mistakes

- 一张表一个聚合，或整个对象图一个聚合。
- 一个用例修改的所有对象组成一个聚合。
- 以 Repository join、事务方便或级联保存作为主要证据。
- 聚合之间持有可直接修改的对象引用。
- 把“看起来属于”当作同步一致性要求。
